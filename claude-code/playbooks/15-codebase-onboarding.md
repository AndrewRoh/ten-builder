# 플레이북 15: 코드베이스 온보딩

> 처음 보는 프로젝트를 AI와 함께 빠르게 파악하는 단계별 가이드

## 언제 쓰나요?

- 새 팀에 합류해서 코드베이스를 처음 읽을 때
- 오픈소스 프로젝트에 기여하려고 구조를 파악할 때
- 6개월 넘게 안 본 내 프로젝트를 다시 만질 때
- 인수인계 없이 레거시 코드를 넘겨받았을 때

## 소요 시간

20-40분

## 사전 준비

- AI 코딩 도구 설치 (Claude Code, Cursor 등)
- 대상 프로젝트 레포 클론 완료
- (선택) 프로젝트 README나 위키 문서

## Step 1: 전체 구조 스캔

코드를 한 줄도 읽기 전에, 먼저 프로젝트의 뼈대를 파악해요.

```bash
# 디렉토리 구조 한눈에 보기 (깊이 2)
tree -L 2 -d --gitignore

# 주요 설정 파일 목록
ls -la package.json pyproject.toml Cargo.toml go.mod Makefile \
  docker-compose.yml .env.example 2>/dev/null
```

AI에게 구조 분석을 요청하면 빠르게 윤곽을 잡을 수 있어요.

```
이 프로젝트의 디렉토리 구조를 분석해줘.
각 폴더의 역할, 사용 중인 프레임워크, 빌드 시스템을 정리해줘.
```

| 확인 항목 | 보는 곳 |
|-----------|---------|
| 언어/프레임워크 | `package.json`, `pyproject.toml`, `go.mod` |
| 진입점 | `main.ts`, `app.py`, `cmd/`, `src/index.*` |
| 빌드 설정 | `Makefile`, `docker-compose.yml`, `CI 파일` |
| 의존성 규모 | `node_modules` 개수, `requirements.txt` 라인 수 |
| 테스트 구조 | `__tests__/`, `test/`, `*_test.go` |

> **팁:** `tree` 출력을 AI에게 그대로 넘기면 분석 정확도가 올라가요. 파일 목록만으로도 아키텍처를 추론할 수 있거든요.

## Step 2: 핵심 파일 5개 읽기

전체 코드를 다 읽을 필요 없어요. 프로젝트의 성격을 결정하는 핵심 파일 5개만 먼저 파악하면 80%는 이해한 거예요.

### 핵심 파일 선정 기준

```
이 프로젝트에서 가장 중요한 파일 5개를 골라줘.
기준: 다른 파일이 가장 많이 import하는 파일, 
비즈니스 로직의 중심 파일, 데이터 모델 정의 파일.
```

```bash
# import가 가장 많이 되는 파일 찾기 (JS/TS)
grep -r "from './" src/ | sed "s/.*from '//;s/'.*//" | sort | uniq -c | sort -rn | head -10

# Python에서 가장 많이 import되는 모듈
grep -r "^from " --include="*.py" | awk -F'from ' '{print $2}' | awk '{print $1}' | sort | uniq -c | sort -rn | head -10
```

### 읽기 순서

1. **데이터 모델** — `models/`, `schema/`, `types/` → 이 프로젝트가 다루는 데이터가 뭔지
2. **진입점** — `main.*`, `app.*`, `index.*` → 앱이 어떻게 시작되는지
3. **라우터/컨트롤러** — `routes/`, `controllers/`, `handlers/` → 외부에 뭘 노출하는지
4. **설정** — `.env.example`, `config/` → 어떤 외부 서비스에 의존하는지
5. **테스트** — 가장 큰 테스트 파일 1개 → 이 프로젝트가 어떻게 동작하는지 실례로 확인

## Step 3: 의존성 지도 그리기

파일 사이의 관계를 파악하면 코드를 수정할 때 영향 범위를 예측할 수 있어요.

```
src/ 디렉토리의 모듈 의존성 그래프를 그려줘.
어떤 모듈이 어떤 모듈을 import하는지 표로 정리해줘.
순환 의존성이 있으면 알려줘.
```

| 분석 대상 | 프롬프트 |
|-----------|---------|
| 모듈 간 관계 | `이 모듈들의 의존 방향을 정리해줘` |
| 순환 참조 | `A→B→C→A 같은 순환 의존성 있어?` |
| 외부 API 연동 | `외부 HTTP 호출이나 SDK 사용 지점을 찾아줘` |
| DB 접근 패턴 | `데이터베이스 쿼리가 어디서 실행돼?` |

```bash
# madge로 의존성 시각화 (JS/TS)
npx madge --image dependency-graph.png src/

# Python import 그래프
pip install pydeps
pydeps src/main.py --max-bacon=2
```

## Step 4: 로컬에서 실행해보기

코드를 읽는 것만으로는 한계가 있어요. 실제로 돌려봐야 동작을 이해할 수 있습니다.

```
이 프로젝트를 로컬에서 실행하려면 어떤 순서로 해야 해?
필요한 환경변수, DB 설정, 빌드 명령어를 정리해줘.
```

### 실행 체크리스트

```bash
# 1. 의존성 설치
npm install  # 또는 pip install -r requirements.txt

# 2. 환경 설정
cp .env.example .env
# .env 파일 편집 — 필요한 값 채우기

# 3. DB 마이그레이션 (있다면)
npm run migrate  # 또는 python manage.py migrate

# 4. 개발 서버 시작
npm run dev  # 또는 python manage.py runserver
```

| 흔한 실행 실패 | 해결 방법 |
|---------------|----------|
| 환경변수 누락 | `.env.example`의 모든 키에 값 채우기 |
| DB 연결 실패 | Docker로 로컬 DB 띄우기 |
| 포트 충돌 | `lsof -i :3000`으로 확인 후 프로세스 종료 |
| 빌드 에러 | Node/Python 버전 확인 (`.nvmrc`, `.python-version`) |

## Step 5: 코드맵 문서 만들기

파악한 내용을 정리해두면 나중에 다시 볼 때, 또는 다른 팀원이 합류할 때 시간을 크게 절약할 수 있어요.

```
지금까지 분석한 내용을 기반으로 이 프로젝트의 코드맵을 만들어줘.
CLAUDE.md 형식으로, 다음 섹션을 포함해서:
- 프로젝트 개요
- 디렉토리 구조와 각 역할
- 핵심 모듈과 의존 관계
- 개발 환경 설정 방법
- 자주 쓰는 명령어
```

### 코드맵 템플릿

```markdown
# 프로젝트 코드맵

## 개요
{한 줄 설명}. {주요 기술 스택}.

## 디렉토리 구조
- `src/api/` — REST 엔드포인트 정의
- `src/services/` — 비즈니스 로직
- `src/models/` — DB 스키마/타입
- `src/utils/` — 공통 유틸리티

## 핵심 흐름
요청 → Router → Controller → Service → Repository → DB

## 개발 명령어
- `npm run dev` — 개발 서버
- `npm test` — 테스트 실행
- `npm run build` — 프로덕션 빌드
```

> **팁:** 이 코드맵을 `CLAUDE.md`로 저장하면 AI 코딩 도구가 매번 프로젝트 컨텍스트를 자동으로 파악해요. 온보딩과 AI 활용을 동시에 해결하는 방법이에요.

## Step 6: 작은 수정으로 감 잡기

구조를 파악했으면, 작은 변경을 직접 해보면서 실전 감각을 키워요.

### 추천 첫 작업

| 난이도 | 작업 | 배우는 것 |
|--------|------|-----------|
| 쉬움 | 오타 수정, 주석 추가 | Git 워크플로우, CI 파이프라인 |
| 보통 | 에러 메시지 개선 | 에러 처리 패턴, 로깅 구조 |
| 보통 | 기존 API에 필드 추가 | 데이터 흐름, 유효성 검사 |
| 어려움 | 버그 수정 | 디버깅 방법, 테스트 패턴 |

```
이 프로젝트에서 "good first issue" 수준의 
개선할 점을 3개 찾아줘.
각각 어떤 파일을 수정해야 하는지도 알려줘.
```
