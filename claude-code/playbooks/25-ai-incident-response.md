# 플레이북 25: AI 장애 대응

> 프로덕션 장애 발생 시 AI 코딩 에이전트로 원인을 빠르게 분석하고 핫픽스를 만드는 단계별 워크플로우

## 언제 쓰나요?

- 프로덕션에서 에러가 급증하고 빠르게 원인을 찾아야 할 때
- 로그가 방대해서 사람이 직접 훑기엔 시간이 부족할 때
- 핫픽스를 최대한 빠르게 배포해야 하지만 실수는 줄여야 할 때
- 장애 포스트모템을 체계적으로 작성하고 싶을 때

## 소요 시간

15-60분 (장애 복잡도에 따라 다름)

## 사전 준비

- Claude Code 또는 AI 코딩 에이전트 설치
- 프로젝트 CLAUDE.md에 모니터링/로깅 관련 컨텍스트 포함
- 로그 접근 가능한 환경 (CLI, 대시보드 등)
- 모니터링 도구 연동 (Grafana, Datadog, Sentry 등)

## 왜 AI 장애 대응이 필요한가

프로덕션 장애에서 가장 많은 시간이 소비되는 구간은 **원인 파악(Root Cause Analysis)**이에요. 평균적으로 전체 대응 시간의 60-70%가 여기에 쓰이죠.

AI 에이전트를 투입하면 이 구간을 크게 줄일 수 있어요:

| 단계 | 수동 대응 | AI 보조 대응 | 절감 효과 |
|------|----------|-------------|----------|
| 로그 분석 | 15-30분 | 2-5분 | 80% |
| 원인 추론 | 10-20분 | 3-8분 | 60% |
| 핫픽스 작성 | 15-30분 | 5-10분 | 65% |
| 영향 범위 파악 | 10-15분 | 1-3분 | 80% |

> 2026년 기준 Rootly, incident.io 같은 AI SRE 도구들이 MTTR(평균 복구 시간)을 40-60% 줄이고 있어요. 같은 원리를 코딩 에이전트에 적용하면 핫픽스까지의 시간을 크게 단축할 수 있습니다.

## Step 1: 장애 인지와 초기 정보 수집

장애가 발생하면 가장 먼저 할 일은 **증상을 정확히 정의**하는 거예요. AI에게 넘기기 전에 맥락을 명확히 정리해야 정확한 분석이 나와요.

### 초기 정보 수집 체크리스트

```markdown
## 장애 요약
- 발생 시각: YYYY-MM-DD HH:MM (TZ)
- 영향 범위: (사용자 수, 서비스명, 지역)
- 증상: (에러 메시지, HTTP 상태코드, 타임아웃 등)
- 최근 배포: (마지막 배포 시각, 변경 내용)
- 재현 가능 여부: (항상/간헐적/특정 조건)
```

### AI에게 컨텍스트 전달

```bash
# 에러 로그를 파일로 추출
kubectl logs deployment/api-server --since=30m > /tmp/incident-logs.txt

# Claude Code에 장애 컨텍스트 전달
claude "프로덕션 장애 분석을 도와줘.

증상: API 서버에서 500 에러 급증 (분당 200건)
발생 시각: 14:30 KST
최근 배포: 14:15에 결제 모듈 업데이트 배포

/tmp/incident-logs.txt 파일을 분석해서:
1. 에러 패턴 분류 (어떤 종류의 에러가 몇 건인지)
2. 에러 발생 시작 시점과 트리거 추정
3. 가장 의심되는 원인 3가지"
```

| 팁 | 설명 |
|----|------|
| 시간 범위 제한 | 로그를 "최근 30분"처럼 범위를 좁혀서 전달 |
| 정상 시점 비교 | 장애 전 로그도 함께 전달하면 차이점 파악이 쉬움 |
| 배포 이력 포함 | 최근 배포 내역은 원인 추론의 핵심 단서 |

## Step 2: AI 기반 원인 분석

초기 정보를 바탕으로 AI에게 체계적인 원인 분석을 요청해요. 핵심은 **가설-검증 루프**를 반복하는 것이에요.

### 가설 생성 프롬프트

```bash
claude "장애 원인 가설을 생성해줘.

수집된 정보:
- 500 에러의 95%가 /api/payments 엔드포인트에서 발생
- 에러 메시지: 'Connection refused to payment-gateway:5432'
- 14:15 배포에서 payment 모듈의 DB 커넥션 설정 변경

가설을 확률 순으로 정리하고,
각 가설을 검증할 수 있는 명령어를 알려줘."
```

### 검증 루프 실행

```bash
# AI가 제안한 검증 명령어를 순서대로 실행
# 가설 1: DB 커넥션 풀 고갈
kubectl exec -it deployment/api-server -- \
  psql -c "SELECT count(*) FROM pg_stat_activity;"

# 가설 2: 새 배포의 환경 변수 누락
kubectl describe deployment/api-server | grep -A5 "Environment"

# 결과를 다시 AI에게 전달
claude "검증 결과:
- pg_stat_activity: 100개 커넥션 (max 100)
- 환경 변수 DB_HOST가 'localhost'로 잘못 설정됨

이 결과를 바탕으로 근본 원인을 확정하고
수정 방향을 제안해줘."
```

### 원인 분석 프레임워크

```
[증상] 500 에러 급증
  ↓
[패턴] 특정 엔드포인트 집중
  ↓
[변경점] 최근 배포에서 설정 변경
  ↓
[근본 원인] 환경 변수 잘못 설정
  ↓
[수정 방향] 올바른 값으로 수정 + 배포
```

## Step 3: 핫픽스 작성

원인이 확정되면 AI에게 **최소 범위의 핫픽스**를 요청해요. 장애 상황에서는 기능 추가나 리팩토링 없이 순수하게 문제만 해결하는 게 중요해요.

### 핫픽스 프롬프트 패턴

```bash
claude "프로덕션 핫픽스를 작성해줘.

근본 원인: payment 서비스의 DB_HOST 환경 변수가
배포 스크립트에서 'localhost'로 하드코딩됨

수정 요구사항:
1. 환경 변수를 ConfigMap에서 읽도록 변경
2. 변경은 해당 파일만, 최소 범위로
3. 기존 테스트가 통과하는지 확인
4. 롤백이 쉬운 구조로 작성

⚠️ 이 PR은 핫픽스입니다. 리팩토링이나 추가 개선은 하지 마세요."
```

### 핫픽스 검증 체크리스트

```bash
# 생성된 핫픽스를 반드시 검증
claude "작성한 핫픽스를 자체 검증해줘:

1. 변경된 파일 목록과 각 변경 이유
2. 기존 동작에 영향을 주는 부분이 있는지
3. 이 수정이 다른 서비스에 사이드 이펙트를 일으킬 가능성
4. 롤백 방법 (이 변경을 되돌리려면 어떻게 해야 하는지)"
```

| 원칙 | 설명 |
|------|------|
| 최소 변경 | 문제 해결에 필요한 코드만 수정 |
| 테스트 포함 | 같은 장애가 재발하지 않도록 테스트 추가 |
| 롤백 가능 | 핫픽스가 더 큰 문제를 일으키면 즉시 되돌릴 수 있어야 함 |
| 주석 포함 | 왜 이렇게 수정했는지 인라인 주석으로 기록 |

## Step 4: 안전한 배포와 모니터링

핫픽스가 준비되면 배포 전에 AI에게 **배포 체크리스트**를 만들어달라고 요청해요.

### 배포 전 체크리스트 생성

```bash
claude "핫픽스 배포 전 체크리스트를 만들어줘:

변경 파일: deploy/payment-service.yaml
변경 내용: DB_HOST를 ConfigMap 참조로 변경

포함할 항목:
1. 배포 전 확인 사항
2. 배포 명령어 (단계별)
3. 배포 후 확인할 메트릭
4. 문제 시 롤백 명령어"
```

### 배포 후 모니터링 프롬프트

```bash
# 배포 후 로그를 실시간 모니터링하면서 AI에게 분석 요청
kubectl logs -f deployment/api-server --since=5m | tee /tmp/post-deploy.txt

# 5분 후 검증
claude "/tmp/post-deploy.txt를 분석해서:
1. 500 에러가 감소했는지 확인
2. 새로운 에러 패턴이 있는지 확인
3. 정상 복구 여부를 판단해줘"
```

## Step 5: 포스트모템 자동 작성

장애가 해결되면 AI에게 **포스트모템 문서**를 작성하게 해요. 장애 대응 중에 수집한 정보를 모두 전달하면 체계적인 문서가 나와요.

### 포스트모템 프롬프트

```bash
claude "장애 포스트모템을 작성해줘.

장애 타임라인:
- 14:15 — 결제 모듈 배포
- 14:30 — 500 에러 급증 알림
- 14:35 — 장애 인지, 원인 분석 시작
- 14:50 — 근본 원인 확정 (DB_HOST 설정 오류)
- 15:00 — 핫픽스 작성 완료
- 15:10 — 핫픽스 배포
- 15:15 — 정상 복구 확인

포스트모템 포함 항목:
1. 요약 (영향 범위, MTTR)
2. 타임라인 (분 단위)
3. 근본 원인 분석
4. 수정 내용
5. 재발 방지 액션 아이템 (담당자, 기한 포함)"
```

### 포스트모템 템플릿

```markdown
# 장애 포스트모템: {제목}

## 요약
- **발생일시:** YYYY-MM-DD HH:MM ~ HH:MM
- **영향 범위:** {사용자 수, 서비스}
- **심각도:** P1 / P2 / P3
- **MTTR:** {분}

## 타임라인
| 시각 | 이벤트 |
|------|--------|
| HH:MM | {이벤트} |

## 근본 원인
{원인 설명}

## 수정 내용
{변경 요약 + PR 링크}

## 재발 방지
| 액션 | 담당 | 기한 |
|------|------|------|
| {액션} | {담당자} | {기한} |
```

## Step 6: 재발 방지 자동화 구축

포스트모템에서 나온 액션 아이템 중 자동화할 수 있는 것을 AI에게 구현 요청해요.

### 자동화 후보 식별

```bash
claude "이 장애에서 배운 교훈을 바탕으로
자동화할 수 있는 방어 수단을 제안해줘:

장애 원인: 배포 시 환경 변수 검증 누락
현재 CI/CD: GitHub Actions

제안 형식:
1. 자동화 항목
2. 구현 방법 (코드 스니펫)
3. 적용 위치 (CI/CD 파이프라인 어디에)"
```

### 방어 수단 예시

```yaml
# .github/workflows/deploy-check.yml
name: Pre-deploy Validation
on:
  pull_request:
    paths: ['deploy/**', 'k8s/**']

jobs:
  validate-env:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check required env vars
        run: |
          # 필수 환경 변수가 ConfigMap에 정의되어 있는지 검증
          for var in DB_HOST DB_PORT DB_NAME; do
            if ! grep -q "$var" deploy/configmap.yaml; then
              echo "Missing required env var: $var"
              exit 1
            fi
          done
```

## 체크리스트

- [ ] 장애 증상과 타임라인을 명확히 정리했다
- [ ] AI에게 로그와 배포 이력을 전달했다
- [ ] 가설-검증 루프를 최소 2회 반복했다
- [ ] 핫픽스는 최소 범위로 작성했다
- [ ] 핫픽스에 재발 방지 테스트를 포함했다
- [ ] 배포 후 모니터링으로 정상 복구를 확인했다
- [ ] 포스트모템 문서를 작성했다
- [ ] 재발 방지 자동화를 구현하거나 이슈로 등록했다

## 자주 하는 실수

| 실수 | 해결 |
|------|------|
| 로그 전체를 AI에게 전달 | 관련 시간 범위로 필터링해서 전달 |
| 핫픽스에 리팩토링 섞기 | 장애 해결만 집중, 개선은 별도 PR |
| 원인 추정만으로 배포 | 반드시 가설을 검증한 후 수정 |
| 포스트모템 생략 | AI가 초안을 만들어주니 빠르게 작성 가능 |
| 1명이 모든 걸 처리 | 분석은 AI, 판단은 사람, 실행은 파이프라인 |
