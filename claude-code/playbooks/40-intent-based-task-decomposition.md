# 플레이북 40: AI 에이전트 인텐트 기반 태스크 분해

> 복잡한 기능 요청을 AI 에이전트가 실행 가능한 단위로 분해하는 체계적 접근법

## 소요 시간

20-30분

## 사전 준비

- Claude Code 또는 Cursor Composer 에이전트 모드 설정 완료
- 프로젝트 CLAUDE.md 또는 .cursorrules 파일 구성
- Git 브랜치 전략 수립 (feature 브랜치 기반 권장)

## 왜 인텐트 기반 분해인가

"로그인 기능 만들어줘"라고 AI에게 한 번에 던지면 컨텍스트 오버플로우, 누락된 엣지 케이스, 일관성 없는 코드가 나온다. 대신 개발자의 의도(인텐트)를 구조화된 단위로 쪼개서 전달하면 결과 품질이 달라진다.

핵심은 **하나의 인텐트 = 하나의 에이전트 세션**이라는 원칙이다.

## Step 1: 인텐트 맵 작성

기능 요청을 받으면 먼저 인텐트 맵을 작성한다. 각 인텐트는 독립적으로 실행 가능한 작업 단위여야 한다.

```markdown
## 인텐트 맵: 사용자 인증 시스템

### 인텐트 1: 데이터 모델 정의
- 범위: User 스키마, 마이그레이션 파일
- 의존성: 없음
- 예상 파일: models/user.ts, migrations/001_create_users.sql

### 인텐트 2: 인증 미들웨어
- 범위: JWT 검증, 세션 관리
- 의존성: 인텐트 1
- 예상 파일: middleware/auth.ts, utils/jwt.ts

### 인텐트 3: API 엔드포인트
- 범위: /login, /register, /logout
- 의존성: 인텐트 1, 2
- 예상 파일: routes/auth.ts, validators/auth.ts

### 인텐트 4: 에러 핸들링 + 테스트
- 범위: 에러 코드 정의, 유닛 테스트
- 의존성: 인텐트 1, 2, 3
- 예상 파일: errors/auth.ts, tests/auth.test.ts
```

| 분해 기준 | 설명 |
|-----------|------|
| 독립 실행 가능 | 다른 인텐트 완료 없이 단독 실행 가능한 것이 이상적 |
| 단일 책임 | 하나의 인텐트가 하나의 관심사만 다룸 |
| 검증 가능 | 완료 여부를 테스트나 빌드로 확인 가능 |
| 컨텍스트 범위 | 관련 파일 5개 이내로 제한 |

## Step 2: 의존성 그래프 그리기

인텐트 간 의존 관계를 파악해서 실행 순서를 결정한다.

```
인텐트 1 (데이터 모델)
    ├── 인텐트 2 (미들웨어)  ← 병렬 가능
    ├── 인텐트 3 (API)       ← 인텐트 2 완료 후
    └── 인텐트 4 (테스트)     ← 인텐트 3 완료 후
```

병렬 실행이 가능한 인텐트는 별도 에이전트 세션(또는 서브에이전트)에 동시에 할당할 수 있다. 이때 중요한 규칙:

```bash
# 병렬 실행 시 git worktree 사용
git worktree add ../project-intent-2 -b intent/auth-middleware
git worktree add ../project-intent-3 -b intent/auth-api

# 각 worktree에서 독립 에이전트 세션 실행
cd ../project-intent-2
claude "인텐트 2: JWT 미들웨어 구현..."

cd ../project-intent-3
claude "인텐트 3: 인증 API 엔드포인트 구현..."
```

## Step 3: 인텐트별 프롬프트 구조화

각 인텐트를 에이전트에 전달할 때 다음 구조를 따른다.

```markdown
## 태스크
{인텐트 제목}

## 컨텍스트
- 프로젝트: {간략 설명}
- 기술 스택: {사용 기술}
- 이전 인텐트에서 생성된 파일: {파일 목록}

## 요구사항
1. {구체적 요구사항 1}
2. {구체적 요구사항 2}
3. {구체적 요구사항 3}

## 제약 조건
- {코딩 컨벤션}
- {성능 기준}
- {보안 요구사항}

## 완료 조건
- [ ] {검증 항목 1}
- [ ] {검증 항목 2}
- [ ] 테스트 통과
- [ ] 타입 에러 없음
```

## Step 4: 실행 + 검증 사이클

각 인텐트 실행 후 반드시 검증 단계를 거친다.

```bash
# 인텐트 완료 후 검증 스크립트
#!/bin/bash

echo "=== 빌드 검증 ==="
npm run build 2>&1 | tail -5

echo "=== 타입 체크 ==="
npx tsc --noEmit 2>&1 | tail -5

echo "=== 테스트 ==="
npm test -- --related 2>&1 | tail -10

echo "=== lint ==="
npm run lint 2>&1 | tail -5
```

| 검증 단계 | 실패 시 대응 |
|-----------|-------------|
| 빌드 실패 | 에이전트에 에러 로그 전달 후 수정 요청 |
| 타입 에러 | 관련 인터페이스 파일과 함께 수정 요청 |
| 테스트 실패 | 실패 테스트 코드 + 구현 코드 함께 전달 |
| lint 경고 | 자동 fix 시도 → 실패 시 수동 수정 |

## Step 5: 인텐트 체인 합치기

모든 인텐트가 완료되면 하나의 feature 브랜치로 합친다.

```bash
# worktree 결과물을 메인 브랜치에 통합
git checkout feature/user-auth
git merge intent/auth-model --no-ff
git merge intent/auth-middleware --no-ff
git merge intent/auth-api --no-ff
git merge intent/auth-tests --no-ff

# 통합 후 전체 검증
npm run build && npm test && npm run lint
```

통합 검증에서 문제가 발견되면, 문제가 있는 인텐트의 에이전트 세션을 다시 열어서 수정한다. 이때 다른 인텐트의 코드를 컨텍스트로 제공하면 호환성 문제를 빠르게 해결할 수 있다.

## 실전 팁

- **인텐트 크기 조절**: 한 인텐트가 파일 5개를 넘기면 더 쪼갠다
- **인터페이스 먼저**: 의존성이 있는 인텐트들은 인터페이스/타입 정의를 먼저 공유한다
- **실패 허용**: 인텐트 하나가 실패해도 독립적인 다른 인텐트는 계속 진행한다
- **메모 남기기**: 각 인텐트 완료 시 CLAUDE.md에 진행 상황을 기록한다

## 체크리스트

- [ ] 기능 요청을 3-5개 인텐트로 분해했는가
- [ ] 각 인텐트의 의존 관계를 파악했는가
- [ ] 병렬 실행 가능한 인텐트를 식별했는가
- [ ] 인텐트별 프롬프트에 컨텍스트와 제약 조건을 포함했는가
- [ ] 완료 조건을 명확히 정의했는가
- [ ] 통합 검증 스크립트를 준비했는가
