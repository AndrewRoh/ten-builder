# 플레이북 24: AI 프롬프트 체이닝

> 복잡한 코딩 태스크를 여러 단계로 쪼개고 순차 실행하는 체이닝 워크플로우

## 언제 쓰나요?

- 한 번의 프롬프트로 처리하기엔 작업 범위가 너무 넓을 때
- "전체 리팩토링해줘" 같은 요청이 매번 엉뚱한 결과를 낼 때
- 분석 → 설계 → 구현 → 검증을 체계적으로 진행하고 싶을 때
- 이전 단계의 출력이 다음 단계의 입력이 되는 파이프라인이 필요할 때

## 소요 시간

15-45분 (태스크 복잡도에 따라 다름)

## 사전 준비

- Claude Code 또는 AI 코딩 에이전트 설치
- 프로젝트 CLAUDE.md 설정 완료
- 작업할 코드베이스 준비

## 왜 체이닝이 필요한가

AI 코딩 에이전트에게 "이 앱 전체를 리팩토링해줘"라고 하면 보통 두 가지 문제가 발생해요:

1. **컨텍스트 과부하** — 모든 걸 한 번에 처리하려다 중요한 부분을 놓침
2. **검증 불가** — 결과물이 한꺼번에 나와서 어디서 잘못됐는지 파악이 어려움

프롬프트 체이닝은 이 문제를 **단계별 분리**로 해결합니다.

| 접근 방식 | 프롬프트 수 | 검증 가능성 | 정확도 |
|-----------|-----------|-----------|--------|
| 한 번에 다 시키기 | 1 | 낮음 | 50-60% |
| 체이닝 (3-5단계) | 3-5 | 단계별 검증 | 85-95% |
| 과도한 분할 (10+) | 10+ | 높지만 비효율 | 90%+ (비용 과다) |

## Step 1: 태스크를 단계로 분해하기

모든 코딩 태스크는 **분석 → 설계 → 구현 → 검증** 패턴으로 분해할 수 있어요. 핵심은 각 단계의 **입력과 출력을 명확하게 정의**하는 것이에요.

### 분해 프레임워크

```
[분석] 현재 상태 파악 + 문제 정의
   ↓ 출력: 문제 목록, 영향 범위
[설계] 해결 방향 결정 + 구조 잡기
   ↓ 출력: 변경 계획, 파일 목록
[구현] 실제 코드 변경
   ↓ 출력: 수정된 파일들
[검증] 테스트 + 리뷰
   ↓ 출력: 통과/실패 리포트
```

### 분해 프롬프트 예시

```bash
# Step 1: 먼저 AI에게 분석만 시킴 (코드 변경 X)
claude "이 프로젝트의 인증 모듈을 분석해줘.
- 현재 인증 방식과 사용 중인 라이브러리
- 보안 취약점이나 개선이 필요한 부분
- 관련 파일 목록과 의존 관계
코드를 수정하지 말고 분석 결과만 정리해줘."
```

> **핵심:** 첫 단계에서는 반드시 "코드를 수정하지 말고"를 명시하세요. AI가 바로 고치려 드는 걸 막아줍니다.

## Step 2: 체이닝 패턴 4가지

### 패턴 A: 순차 파이프라인 (가장 기본)

한 단계의 출력이 다음 단계의 입력이 되는 구조예요.

```bash
# Stage 1: 분석
claude "src/auth/ 디렉토리의 코드를 분석해서 개선 포인트를 파일로 정리해줘.
결과를 /tmp/auth-analysis.md로 저장해."

# Stage 2: 설계 (이전 분석 결과를 참조)
claude "/tmp/auth-analysis.md를 읽고, 각 개선 포인트별로 구체적인 변경 계획을 세워줘.
파일별로 뭘 바꿔야 하는지, 순서는 어떻게 되는지 정리해서
/tmp/auth-plan.md로 저장해."

# Stage 3: 구현 (계획대로 실행)
claude "/tmp/auth-plan.md의 1번 항목만 구현해줘.
나머지는 건드리지 마."
```

| 장점 | 주의점 |
|------|--------|
| 각 단계에서 결과 확인 가능 | 중간 파일 관리 필요 |
| 문제 발생 시 해당 단계만 재실행 | 단계 간 맥락 손실 가능 |
| 복잡도 조절 용이 | 전체 시간은 늘어남 |

### 패턴 B: Fan-out / Fan-in (병렬 처리)

독립적인 작업을 여러 세션에서 동시에 처리하고 결과를 합치는 패턴이에요.

```bash
# Fan-out: 3개 작업을 병렬로
claude "src/api/routes.ts의 에러 핸들링을 개선해줘" &
claude "src/api/middleware.ts의 로깅을 구조화해줘" &
claude "src/api/validators.ts의 입력 검증을 강화해줘" &
wait

# Fan-in: 결과를 통합 검증
claude "방금 변경된 src/api/ 디렉토리의 routes.ts, middleware.ts,
validators.ts를 전체적으로 리뷰해줘.
서로 충돌하는 부분이 있는지, 통합 테스트가 필요한지 확인해."
```

> **주의:** Fan-out은 각 작업이 서로 다른 파일을 수정할 때만 안전합니다. 같은 파일을 동시에 건드리면 충돌이 발생해요.

### 패턴 C: 조건 분기 (Gate 패턴)

이전 단계의 결과에 따라 다음 행동이 달라지는 패턴이에요.

```bash
# Stage 1: 코드 분석 + 판단
claude "src/database/ 디렉토리의 쿼리 성능을 분석해줘.
- N+1 쿼리 문제가 있는지
- 인덱스가 필요한 컬럼이 있는지
- 개선이 필요하면 'NEEDS_FIX'를, 괜찮으면 'LOOKS_GOOD'을
/tmp/db-check.txt에 한 줄로 적어줘."

# Stage 2: 조건 분기
STATUS=$(cat /tmp/db-check.txt | head -1)
if [[ "$STATUS" == *"NEEDS_FIX"* ]]; then
  claude "src/database/의 N+1 쿼리를 JOIN으로 변환하고 인덱스 마이그레이션을 생성해줘."
else
  echo "DB 쿼리 상태 양호 — 스킵"
fi
```

### 패턴 D: 반복 개선 (Loop 패턴)

테스트를 통과할 때까지 구현 → 검증을 반복하는 패턴이에요.

```bash
MAX_ATTEMPTS=3
ATTEMPT=1

while [ $ATTEMPT -le $MAX_ATTEMPTS ]; do
  echo "=== 시도 $ATTEMPT/$MAX_ATTEMPTS ==="

  # 구현
  claude "src/utils/parser.ts의 에지 케이스를 처리해줘.
  이전 테스트 실패 로그가 있으면 참고해: /tmp/test-failures.log"

  # 검증
  if npm test -- --testPathPattern=parser 2>/tmp/test-failures.log; then
    echo "테스트 통과!"
    break
  fi

  ATTEMPT=$((ATTEMPT + 1))
done
```

## Step 3: 실전 시나리오 — REST API 리팩토링

실제로 프롬프트 체이닝을 적용하는 예시를 볼게요. 기존 Express API를 구조화하는 작업이에요.

### Chain 1: 현재 상태 감사

```bash
claude "이 Express 프로젝트의 API 구조를 감사해줘.

확인할 것:
1. 라우트 목록과 HTTP 메서드
2. 미들웨어 사용 현황
3. 에러 핸들링 패턴 (일관성 여부)
4. 응답 포맷 통일 여부

분석 결과를 docs/api-audit.md에 저장해.
코드는 수정하지 마."
```

### Chain 2: 리팩토링 계획 수립

```bash
claude "docs/api-audit.md를 기반으로 리팩토링 계획을 세워줘.

규칙:
- 한 번에 하나의 패턴만 변경
- 각 변경의 영향 범위 명시
- 테스트 방법 포함
- 우선순위: 에러 핸들링 > 응답 포맷 > 미들웨어

결과를 docs/refactor-plan.md에 저장해."
```

### Chain 3: 단계별 구현

```bash
claude "docs/refactor-plan.md의 1단계(에러 핸들링 통일)만 구현해줘.
- 글로벌 에러 핸들러 미들웨어 생성
- 커스텀 에러 클래스 정의
- 기존 try-catch를 새 패턴으로 전환
구현 후 관련 테스트를 실행해서 결과를 알려줘."
```

### Chain 4: 검증 + 다음 단계

```bash
claude "1단계 변경사항을 리뷰해줘.
- 에러 응답 포맷이 일관적인지
- 기존 API 계약(status code, 응답 구조)이 깨지지 않았는지
- 빠뜨린 라우트가 없는지
문제가 없으면 docs/refactor-plan.md의 2단계로 넘어가."
```

## Step 4: 체이닝 자동화 — 슬래시 커맨드 활용

반복적으로 사용하는 체인은 커맨드로 등록해두면 편리해요.

### .claude/commands/chain-refactor.md

```markdown
# 체이닝 리팩토링

$ARGUMENTS를 순차적으로 리팩토링합니다.

## 절차

1. 대상 디렉토리의 코드를 분석해서 /tmp/chain-analysis.md에 저장
2. 분석 결과를 기반으로 변경 계획을 /tmp/chain-plan.md에 저장
3. 계획의 각 항목을 순서대로 구현 (한 항목씩)
4. 각 구현 후 테스트 실행
5. 테스트 실패 시 해당 항목만 롤백하고 다음으로

## 규칙
- 한 번에 하나의 파일 그룹만 변경
- 각 단계 완료 후 git diff로 변경 확인
- 전체 완료 후 통합 테스트 실행
```

```bash
# 사용
claude "/chain-refactor src/controllers/"
```

### .claude/commands/chain-test-fix.md

```markdown
# 테스트 수정 체인

실패하는 테스트를 분석하고 수정합니다.

1. 실패 테스트 로그를 분석해서 원인 분류
2. 각 원인별 수정 계획 수립
3. 수정 → 테스트 → 통과 확인을 반복
4. 최대 3회 시도 후 실패하면 사람에게 에스컬레이션
```

## Step 5: 체이닝 설계 원칙

### 좋은 체인의 특징

```
[명확한 입력] → [단일 책임] → [검증 가능한 출력]
```

| 원칙 | 설명 | 예시 |
|------|------|------|
| 단일 책임 | 한 단계에서 한 가지만 처리 | 분석 단계에서 코드 수정 X |
| 명시적 출력 | 각 단계의 결과물을 파일로 저장 | `/tmp/analysis.md`로 저장 |
| 검증 게이트 | 다음 단계 진입 전 결과 확인 | 테스트 통과 여부 체크 |
| 롤백 가능 | 실패 시 이전 상태로 복구 | `git stash` 또는 브랜치 활용 |
| 점진적 복잡도 | 쉬운 것부터 어려운 것 순서로 | 타입 에러 → 로직 에러 → 아키텍처 |

### 흔한 실수와 해결

| 실수 | 문제 | 해결 |
|------|------|------|
| 단계를 너무 잘게 쪼갬 | 컨텍스트 전환 비용 증가 | 3-5단계가 적정 |
| 단계 간 맥락 누락 | AI가 이전 결정을 잊음 | 중간 결과를 파일로 전달 |
| 검증 없이 다음으로 | 에러가 누적됨 | 매 단계 후 테스트/리뷰 |
| 전체를 자동화 | 중간에 사람 판단 필요 | 핵심 분기점에 수동 확인 |

## Step 6: 컨텍스트 전달 기법

체이닝에서 가장 중요한 건 **단계 간 맥락을 잃지 않는 것**이에요.

### 방법 1: 파일 기반 전달

```bash
# 분석 결과를 파일로 저장
claude "분석 결과를 docs/context/step1-analysis.md에 저장해."

# 다음 단계에서 명시적으로 참조
claude "docs/context/step1-analysis.md를 읽고 이를 기반으로 작업해."
```

### 방법 2: CLAUDE.md에 진행 상황 기록

```markdown
# 프로젝트 상태

## 현재 진행 중인 리팩토링
- [x] Step 1: API 감사 완료 (docs/api-audit.md 참조)
- [x] Step 2: 에러 핸들링 통일 완료
- [ ] Step 3: 응답 포맷 표준화 진행 중
- [ ] Step 4: 미들웨어 정리

## 결정 사항
- 에러 포맷: { error: { code, message, details } }
- HTTP 상태 코드: RFC 7231 준수
```

### 방법 3: Git 커밋을 체크포인트로 활용

```bash
# 각 체인 단계마다 커밋
git add -A && git commit -m "refactor(step1): error handling 통일"

# 문제 발생 시 특정 단계로 롤백
git revert HEAD  # 마지막 단계만 취소
```

## 체크리스트

- [ ] 태스크를 3-5개 단계로 분해했는가
- [ ] 각 단계의 입력/출력이 명확한가
- [ ] 첫 단계에서 "코드 수정하지 말 것"을 명시했는가
- [ ] 단계 간 맥락 전달 방법을 정했는가 (파일/CLAUDE.md/커밋)
- [ ] 검증 게이트가 설정되어 있는가 (테스트, 리뷰)
- [ ] 실패 시 롤백 방법이 있는가
