# 가이드 19: AI 태스크 분해

> AI 코딩 에이전트에게 작업을 잘 맡기려면 "어떻게 쪼갤 것인가"가 핵심이에요

## 소요 시간

20-30분

## 사전 준비

- AI 코딩 도구 사용 경험 (Claude Code, Cursor, Copilot 등)
- 프로젝트 하나 이상 개발 경험
- 기본적인 Git 워크플로우 이해

## 왜 태스크를 쪼개야 하나요?

AI 에이전트에게 "이 앱 만들어줘"라고 하면 높은 확률로 어딘가 빠지거나 엉뚱한 방향으로 가요. 반면 "이 API 엔드포인트에 입력 검증 미들웨어를 추가해줘"라고 하면 거의 완벽한 결과가 나와요.

차이점은 **태스크의 범위**예요.

```
❌ 큰 태스크 (실패 확률 높음)
"이커머스 사이트를 풀스택으로 만들어줘"

✅ 분해된 태스크 (성공 확률 높음)
1. "상품 스키마를 설계해줘 (name, price, category, stock)"
2. "GET /products 엔드포인트를 만들어줘 (페이지네이션 포함)"
3. "상품 목록 컴포넌트를 만들어줘 (카드 레이아웃)"
```

핵심: **컨텍스트 윈도우가 아무리 넓어도, 태스크 범위가 넓으면 결과 품질이 떨어져요.** AI 에이전트는 좁고 명확한 작업에서 최고 성능을 발휘해요.

## Step 1: 태스크 크기 판단 기준

작업을 받았을 때 먼저 "이걸 한 번에 시킬 수 있는가?"를 판단해야 해요.

| 기준 | 한 번에 가능 | 분해 필요 |
|------|-------------|----------|
| **파일 수** | 1-3개 변경 | 4개 이상 변경 |
| **관심사** | 단일 관심사 | 여러 레이어 동시 변경 |
| **의존성** | 독립적 | 다른 변경에 의존 |
| **검증** | 테스트 하나로 확인 | 여러 종류의 테스트 필요 |
| **설명 길이** | 2-3문장 | 문단 이상 |

실전 팁: **PR 하나에 들어갈 수 있는 크기**가 좋은 기준이에요. 리뷰어가 10분 안에 이해할 수 있는 변경이면 적당해요.

## Step 2: 분해 패턴 4가지

### 패턴 1: 레이어별 분해 (Vertical Slicing)

풀스택 작업을 레이어별로 나눠요.

```
전체 태스크: "사용자 프로필 편집 기능 추가"

분해:
① DB 스키마 + 마이그레이션
② API 엔드포인트 (CRUD)
③ 프론트엔드 폼 컴포넌트
④ 입력 검증 + 에러 처리
⑤ 테스트 작성
```

```bash
# 각 단계를 별도 세션에서 진행
# ① 스키마 작업
claude "users 테이블에 bio, avatar_url, social_links 컬럼을 추가하는 마이그레이션을 만들어줘"

# ② API 작업 (①의 결과물을 참조)
claude "UserProfile 모델 기반으로 GET/PUT /api/profile 엔드포인트를 만들어줘"
```

### 패턴 2: 기능별 분해 (Feature Slicing)

하나의 기능 안에서 세부 기능을 나눠요.

```
전체 태스크: "검색 기능 구현"

분해:
① 기본 텍스트 검색 (exact match)
② 퍼지 검색 (typo tolerance)
③ 필터링 (카테고리, 날짜, 태그)
④ 검색 결과 하이라이팅
⑤ 최근 검색어 저장
```

이 패턴의 장점은 **각 단계마다 동작하는 코드**가 나온다는 거예요. ①만 끝나도 기본 검색은 작동해요.

### 패턴 3: Plan-Act-Reflect (계획-실행-검토)

복잡한 작업에 적합한 3단계 프레임워크예요.

```
[Plan] "이 코드베이스를 분석하고 리팩토링 계획을 세워줘"
  → 에이전트가 구조를 파악하고 변경 목록 작성

[Act]  "계획의 1번 항목을 실행해줘"
  → 실제 코드 변경

[Reflect] "변경된 코드를 리뷰하고 문제점을 찾아줘"
  → 에이전트가 자기 결과물을 검증
```

```bash
# Plan 단계
claude "src/ 디렉토리의 구조를 분석하고,
중복 코드와 개선 가능한 부분을 목록으로 정리해줘.
코드를 수정하지 말고 분석만 해줘."

# Act 단계 (Plan 결과를 참조)
claude "분석 결과의 1번 '유틸 함수 중복 제거'를 실행해줘.
utils/string.ts와 helpers/format.ts의 중복 함수를 하나로 합쳐줘."

# Reflect 단계
claude "방금 변경한 파일들의 import가 모두 정상인지 확인하고,
빠진 테스트가 있으면 추가해줘."
```

### 패턴 4: 스펙 기반 분해 (Spec-Driven)

작업 사양서를 먼저 만들고, 그걸 기반으로 구현을 나눠요.

```markdown
## 태스크 스펙: 알림 시스템

### 입력
- 이벤트 타입: order_placed, payment_failed, item_shipped
- 수신자: userId, email
- 채널: in-app, email, push

### 출력
- 알림 저장 (DB)
- 채널별 발송

### 제약 조건
- 중복 발송 방지 (idempotency key)
- 실패 시 3회 재시도
- 발송 지연: 최대 30초

### 구현 순서
1. Notification 모델 + 저장 로직
2. In-app 채널 구현
3. Email 채널 구현
4. 재시도 로직
5. Idempotency 처리
```

스펙을 에이전트에게 전체 공유한 뒤, 구현 순서대로 하나씩 맡겨요. 에이전트가 전체 그림을 알면서도 좁은 범위에 집중할 수 있어요.

## Step 3: 에이전트에게 전달하는 구조화된 지시

태스크를 쪼갰으면, 각 조각을 에이전트에게 잘 전달해야 해요.

### 좋은 태스크 지시의 5요소

```markdown
## 태스크: [한 줄 요약]

### 컨텍스트
- 이 프로젝트는 [프로젝트 설명]
- 현재 [관련 파일/모듈] 상태: [현재 상태]

### 목표
- [구체적인 결과물 1]
- [구체적인 결과물 2]

### 제약 조건
- [기술 스택/라이브러리 제한]
- [코딩 스타일/패턴 준수]
- [성능/보안 요구사항]

### 완료 기준
- [ ] [테스트 통과 여부]
- [ ] [타입 체크 통과]
- [ ] [특정 기능 동작 확인]
```

### 나쁜 지시 vs 좋은 지시

```bash
# ❌ 나쁜 예
claude "인증 기능 만들어줘"

# ✅ 좋은 예
claude "다음 요구사항으로 JWT 인증 미들웨어를 만들어줘:
- express 미들웨어로 구현
- Authorization: Bearer <token> 헤더에서 토큰 추출
- jsonwebtoken 패키지 사용 (이미 설치됨)
- 토큰 검증 실패 시 401 반환
- 검증 성공 시 req.user에 디코딩된 페이로드 저장
- 테스트: 유효/만료/없는 토큰 3가지 케이스"
```

## Step 4: 태스크 체이닝 기법

쪼갠 태스크들을 순서대로 연결하는 방법이에요.

### 선형 체이닝

이전 결과를 다음 태스크의 입력으로 넘겨요.

```bash
# 태스크 1 → 결과물 확인 → 태스크 2 → 결과물 확인 → ...
claude "User 모델을 만들어줘 (id, email, name, createdAt)"
# 결과 확인 후
claude "방금 만든 User 모델 기반으로 회원가입 API를 만들어줘"
# 결과 확인 후
claude "회원가입 API에 이메일 중복 검사를 추가해줘"
```

### 병렬 체이닝

독립적인 태스크를 동시에 실행해요.

```bash
# 서로 의존성 없는 작업은 병렬로
# 터미널 1: API 엔드포인트
claude "GET /api/stats 엔드포인트를 만들어줘"

# 터미널 2: 프론트 컴포넌트 (별도 세션)
claude "대시보드 통계 카드 컴포넌트를 만들어줘"

# 나중에 합치기
claude "Stats 컴포넌트에서 /api/stats를 호출하도록 연결해줘"
```

### 의존성 그래프

복잡한 프로젝트는 의존성을 그래프로 그려서 실행 순서를 정해요.

```
① 모델 정의 ─────────┐
                      ├── ④ API 엔드포인트 ── ⑥ 통합 테스트
② 유틸 함수 ──────────┘
                      ┌── ⑤ UI 컴포넌트 ─── ⑥ 통합 테스트
③ 디자인 시스템 확인 ──┘
```

①②③은 병렬 실행 가능, ④⑤는 선행 작업 완료 후 실행, ⑥은 모두 완료 후 실행.

## Step 5: 실전 분해 사례

### 사례: "블로그에 댓글 기능 추가"

```
1일차 (기반)
├── T1: Comment 모델 + DB 마이그레이션
├── T2: POST /api/comments 엔드포인트
└── T3: GET /api/posts/:id/comments 엔드포인트

2일차 (프론트)
├── T4: 댓글 목록 컴포넌트
├── T5: 댓글 작성 폼
└── T6: 댓글 삭제 (본인 댓글만)

3일차 (고도화)
├── T7: 대댓글 (1단계 깊이만)
├── T8: 댓글 수 표시 (게시물 목록)
└── T9: 스팸 필터 (기본 키워드 차단)
```

각 태스크는:
- 독립적으로 테스트 가능
- PR 하나에 담을 수 있는 크기
- 실패해도 다른 태스크에 영향 최소화

## 자주 하는 실수와 해결법

| 실수 | 해결 |
|------|------|
| 한 번에 너무 많이 시키기 | 파일 3개 이내 변경으로 제한 |
| 컨텍스트 없이 지시하기 | 관련 파일/모듈 상태를 먼저 알려주기 |
| 완료 기준을 생략하기 | "이 테스트가 통과하면 완료"를 명시 |
| 모든 걸 순서대로 하기 | 독립적인 태스크는 병렬로 실행 |
| Plan 없이 바로 Act | 복잡한 작업은 분석 먼저, 구현은 나중에 |
| 검토(Reflect) 건너뛰기 | 각 단계 결과물 확인 후 다음 진행 |

## 태스크 크기 체크리스트

- [ ] 한 문장으로 설명할 수 있는가?
- [ ] 변경 파일이 3개 이하인가?
- [ ] 완료 기준이 명확한가?
- [ ] 10분 이내에 결과를 확인할 수 있는가?
- [ ] 실패해도 되돌리기 쉬운가?

전부 체크되면 에이전트에게 바로 맡겨도 돼요. 하나라도 빠지면 더 쪼개야 해요.
