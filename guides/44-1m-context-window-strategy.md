# 가이드 44: 1M 컨텍스트 윈도우 실전 활용 전략

> 대규모 코드베이스를 통째로 넣을 수 있게 됐다 — 그런데 정말 그렇게 쓰는 게 맞을까?

## 왜 1M 컨텍스트가 중요한가

Claude Code Opus 4.6부터 1M 토큰 컨텍스트 윈도우가 표준 가격으로 제공됩니다. 이전에는 128K~200K 제한으로 대규모 코드베이스를 여러 세션에 나눠 분석해야 했지만, 이제 중간 규모 프로젝트 전체를 한 번에 로드할 수 있어요.

하지만 "넣을 수 있다"와 "넣어야 한다"는 다릅니다. 이 가이드에서는 1M 컨텍스트를 실제로 효과적으로 쓰는 패턴과, 오히려 성능이 떨어지는 안티패턴을 정리합니다.

## 1M 토큰으로 할 수 있는 것

| 항목 | 대략적 분량 | 활용 시나리오 |
|------|------------|-------------|
| TypeScript 파일 | ~2,000개 (500줄 평균) | 전체 프론트엔드 프로젝트 분석 |
| Python 패키지 | ~1,500개 모듈 | Django/FastAPI 전체 코드베이스 |
| 문서 + 코드 혼합 | README 50개 + 코드 1,000개 | 모노레포 아키텍처 이해 |
| Git diff | ~10,000줄 변경분 | 대규모 리팩토링 리뷰 |

## 실전 활용 패턴 5가지

### 패턴 1: 전체 아키텍처 맵핑

프로젝트에 처음 투입됐을 때, 코드베이스 전체를 컨텍스트에 넣고 아키텍처 다이어그램을 요청하세요.

```bash
# 프로젝트 전체 파일을 한 번에 분석
find src/ -name '*.ts' -o -name '*.tsx' | head -500 | \
  xargs cat | wc -c
# → 대략 2MB 이하면 1M 토큰 안에 들어감

# Claude Code에서
claude "이 프로젝트의 전체 아키텍처를 분석해줘. 
모듈 간 의존성, 데이터 흐름, 진입점을 정리해줘."
```

**왜 효과적인가:** 파일을 하나씩 보면 놓치는 크로스 모듈 의존성을 한 번에 파악할 수 있습니다.

### 패턴 2: 크로스 파일 리팩토링 영향 분석

함수 시그니처를 변경할 때, 호출하는 모든 파일을 컨텍스트에 포함시켜 영향 범위를 정확히 파악합니다.

```bash
# 특정 함수를 사용하는 모든 파일 찾기
grep -rl "processPayment" src/ | xargs cat > /tmp/context.txt
wc -c /tmp/context.txt  # 토큰 수 확인

# Claude Code에서
claude "processPayment 함수의 반환 타입을 
Promise<PaymentResult>에서 Promise<Result<PaymentResult, PaymentError>>로 
변경하려고 해. 영향받는 모든 파일과 수정 계획을 알려줘."
```

### 패턴 3: 레거시 코드 전체 마이그레이션 계획

레거시 프로젝트의 모든 파일을 넣고, 모던 스택으로의 마이그레이션 전략을 한 번에 수립합니다.

```bash
# 레거시 코드 + 새 프로젝트 보일러플레이트를 함께 넣기
claude "legacy/ 디렉토리의 jQuery 코드를 React로 마이그레이션하는 
단계별 계획을 세워줘. 컴포넌트 분리 기준과 우선순위를 포함해."
```

### 패턴 4: 전체 테스트 스위트 분석

테스트 파일 전체를 넣고 커버리지 갭과 중복 테스트를 한 번에 식별합니다.

```bash
# 테스트 파일 전체 + 소스 코드를 함께 분석
claude "tests/ 디렉토리의 모든 테스트와 src/를 비교해서:
1. 테스트가 없는 함수/모듈 목록
2. 중복되는 테스트 케이스
3. 엣지케이스가 부족한 부분을 정리해줘."
```

### 패턴 5: PR 메가 리뷰

대규모 PR(1,000줄+)의 diff 전체를 컨텍스트에 넣고 종합 리뷰를 받습니다.

```bash
# GitHub PR diff 추출
gh pr diff 42 --repo myorg/myrepo > /tmp/pr-diff.txt

# 원본 코드와 함께 분석
claude "이 PR diff를 리뷰해줘. 
보안 취약점, 성능 이슈, 아키텍처 위반을 중점적으로 봐줘.
원본 파일도 함께 참고해."
```

## 주의할 안티패턴 3가지

### 안티패턴 1: 무조건 전부 넣기

```bash
# 나쁜 예: node_modules까지 포함
find . -name '*.js' | xargs cat  # ← 토큰 낭비
```

**대신:** `src/`, `lib/` 등 프로젝트 코드만 선별적으로 포함하세요. 의존성 코드는 타입 정의(`.d.ts`)만 넣으면 됩니다.

### 안티패턴 2: 단순 작업에 대용량 컨텍스트

단일 파일 버그 수정이나 간단한 유틸 함수 작성에는 1M 컨텍스트가 필요 없습니다. 오히려 응답 시간만 늘어나요.

| 작업 유형 | 권장 컨텍스트 크기 |
|----------|-----------------|
| 단일 파일 수정 | 해당 파일 + 관련 타입 (~10K) |
| 모듈 내 리팩토링 | 모듈 전체 (~50K) |
| 크로스 모듈 변경 | 관련 모듈 + 인터페이스 (~200K) |
| 전체 아키텍처 분석 | 프로젝트 전체 (~500K+) |

### 안티패턴 3: 컨텍스트만 믿고 검증 안 하기

1M 컨텍스트라도 주의력은 균일하지 않습니다. 중간 부분의 정보가 처음이나 끝보다 덜 정확하게 처리될 수 있어요 (Lost in the Middle 현상).

**대응 방법:**
- 중요한 파일은 컨텍스트 앞쪽에 배치
- 분석 결과를 실제 코드와 교차 검증
- 전체 분석 후 부분별로 다시 확인

## 비용 최적화 전략

1M 토큰을 매번 채우면 비용이 빠르게 증가합니다.

```
# 비용 계산 예시 (Claude Opus 4.6 기준)
입력: 1M 토큰 × $15/1M = $15/세션
출력: 평균 4K 토큰 × $75/1M = $0.30/세션
총 비용: ~$15.30/세션
```

**절약 팁:**
1. **프롬프트 캐싱 활용:** 동일 코드베이스 반복 분석 시 캐시 히트로 90% 절감
2. **계층적 접근:** 먼저 디렉토리 구조만 분석 → 필요한 부분만 상세 분석
3. **Haiku로 필터링:** 간단한 검색/필터는 Haiku, 심층 분석만 Opus 사용

## 실전 워크플로우: 새 프로젝트 온보딩

```
Step 1: 디렉토리 구조 + README만 넣기 (5K 토큰)
   → 프로젝트 개요 파악

Step 2: 핵심 모듈 코드 추가 (100K 토큰)
   → 아키텍처 패턴 이해

Step 3: 전체 소스 + 테스트 (500K+ 토큰)
   → 크로스 의존성 분석, 리팩토링 계획

Step 4: 변경 대상 파일만 집중 (50K 토큰)
   → 실제 코드 수정
```

이렇게 단계별로 컨텍스트를 확장하면 비용은 절약하면서도 필요할 때 전체 맥락을 활용할 수 있습니다.
