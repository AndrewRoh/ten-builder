# 가이드 21: 팀에 AI 코딩 도구 도입하기

> 혼자 쓰면 편하고, 팀으로 가면 혼란스럽다 — 체계적 도입이 답이에요

## 소요 시간

30-40분

## 사전 준비

- AI 코딩 도구 개인 사용 경험 (Claude Code, Cursor, Copilot 등)
- 팀 리드 이상 직급이거나, 팀 내 도구 도입 제안 가능한 위치
- Git 기반 협업 워크플로우 운영 경험

## 왜 팀 단위 도입이 다른가요?

혼자 AI 도구를 쓰면 자유롭게 실험하고 자기만의 방식을 만들 수 있어요. 하지만 팀에서는 달라요.

```
혼자 쓸 때:
  나 → AI 도구 → 코드

팀에서 쓸 때:
  팀원 A → AI 도구 A → 코드 → PR 리뷰 → 머지
  팀원 B → AI 도구 B → 코드 → PR 리뷰 → 머지
  팀원 C → 수동 작성 → 코드 → PR 리뷰 → 머지
  
  → 스타일 불일치, 리뷰 기준 혼란, 품질 편차
```

팀 도입의 핵심은 **"모두가 같은 기준으로 AI를 쓰는 것"**이에요. 도구 선택이 아니라 **프로세스 설계**가 진짜 과제예요.

## Step 1: 현재 팀 상태 파악

도입 전에 팀의 현재 위치를 파악해요. 무작정 도구를 던져주면 저항만 커져요.

### AI 도입 성숙도 체크리스트

| 레벨 | 설명 | 특징 |
|------|------|------|
| **L0 — 미도입** | AI 도구 사용 안 함 | "기존 방식이 충분하다" |
| **L1 — 개인 실험** | 일부 팀원이 개인적으로 사용 | Copilot 자동완성 수준 |
| **L2 — 부분 도입** | 팀 내 AI 사용 가이드라인 존재 | 특정 작업에만 허용 |
| **L3 — 표준화** | 팀 전체 도구/프로세스 통일 | CLAUDE.md, 코드 리뷰 기준 포함 |
| **L4 — 최적화** | 생산성 측정 + 지속 개선 | 메트릭 기반 피드백 루프 |

대부분의 팀은 L0~L1이에요. L3까지 가는 게 이 가이드의 목표예요.

### 팀원별 유형 분석

```
📊 팀원 유형별 접근법:

얼리어답터 (20%)  → 챔피언으로 활용. 사례 공유 역할
관망자 (50%)      → 실제 효과를 보여주면 전환. 데모가 유효
회의론자 (20%)    → 품질/보안 우려. 데이터로 설득
거부자 (10%)      → 강제하지 않음. 자연스럽게 노출
```

## Step 2: 도입 로드맵 설계

한 번에 전부 바꾸려 하면 실패해요. 4주 단위 점진적 도입이 현실적이에요.

### Phase 1: 파일럿 (1-2주)

**목표:** 2-3명의 얼리어답터가 실제 프로젝트에서 AI 도구 검증

```yaml
# 파일럿 팀 구성
participants:
  - role: 시니어 개발자 1명   # 코드 품질 기준 설정
  - role: 미드레벨 1명        # 실제 생산성 변화 측정
  - role: 주니어 1명          # 온보딩/학습 효과 확인

scope:
  - 신규 기능 개발 1건
  - 기존 코드 리팩토링 1건
  - 테스트 코드 추가 1건

output:
  - 파일럿 결과 보고서
  - 팀 가이드라인 초안
  - 도구 비용 추정
```

### Phase 2: 가이드라인 수립 (1주)

파일럿 결과를 바탕으로 팀 규칙을 만들어요.

```markdown
# AI 코딩 도구 팀 가이드라인 (예시)

## 허용 범위
- ✅ 코드 자동완성 (Copilot, Cursor Tab)
- ✅ 테스트 코드 생성 (반드시 사람이 검증)
- ✅ 보일러플레이트 코드 생성
- ✅ 문서/README 초안 작성
- ⚠️ 비즈니스 로직 작성 (반드시 PR 리뷰)
- ❌ 보안 관련 코드 전적 위임
- ❌ 검증 없이 AI 생성 코드 머지

## PR 리뷰 규칙
- AI 생성 코드 비율이 높은 PR은 반드시 2명 리뷰
- 리뷰어는 "AI가 생성한 부분" 표시 요청 가능
- 테스트 커버리지 기준은 AI/수동 동일 적용

## 프로젝트 설정
- 모든 프로젝트에 CLAUDE.md (또는 동급 설정 파일) 필수
- .cursorrules, .github/copilot-instructions.md 통일
```

### Phase 3: 팀 전체 롤아웃 (2주)

```bash
# 롤아웃 체크리스트
- [ ] 팀 전원 도구 설치 완료
- [ ] CLAUDE.md 프로젝트 루트에 추가
- [ ] AI 코딩 가이드라인 위키에 공유
- [ ] PR 템플릿에 AI 사용 여부 체크박스 추가
- [ ] 주간 회고에 AI 관련 항목 포함
```

### Phase 4: 측정 + 개선 (지속)

도입하고 끝이 아니에요. 측정해야 개선할 수 있어요.

## Step 3: 생산성 측정 방법

"AI 쓰니까 빨라졌다"는 감이 아니라 데이터가 필요해요.

### 추천 메트릭 (GAINS 프레임워크 참고)

| 메트릭 | 측정 방법 | 주의점 |
|--------|----------|--------|
| **PR 사이클 타임** | 첫 커밋 → 머지까지 소요 시간 | 리뷰 대기 시간 분리 필요 |
| **일일 완료 태스크** | Jira/Linear 기준 완료 이슈 수 | 태스크 크기 표준화 필요 |
| **테스트 커버리지 변화** | CI 리포트 추적 | AI 생성 테스트 품질 별도 확인 |
| **버그 발생률** | 릴리즈 후 버그 / 전체 커밋 비율 | AI 코드 vs 수동 코드 구분 |
| **개발자 만족도** | 월 1회 설문 (1-5 스케일) | 익명성 보장 필수 |

### 실전 측정 팁

```bash
# GitHub API로 PR 사이클 타임 자동 추적 (예시)
gh api repos/{owner}/{repo}/pulls?state=closed \
  --jq '.[] | {
    title: .title,
    created: .created_at,
    merged: .merged_at,
    cycle_hours: ((.merged_at | fromdateiso8601) - (.created_at | fromdateiso8601)) / 3600
  }' | head -20
```

비교 기간은 최소 4주예요. 1-2주는 학습 곡선 때문에 오히려 느려질 수 있어요.

## Step 4: 저항 극복 전략

### 흔한 반대 의견과 대응

| 반대 의견 | 핵심 우려 | 대응 방법 |
|----------|----------|----------|
| "AI 코드는 품질이 낮다" | 코드 품질 | 파일럿 결과 데이터 공유 + 리뷰 프로세스 강화 |
| "내 직업이 위험해진다" | 고용 불안 | AI는 도구이지 대체재가 아님. 더 복잡한 문제에 집중 가능 |
| "보안 위험이 있다" | 정보 유출 | 기업용 플랜, 로컬 모델, 데이터 정책 수립 |
| "배울 시간이 없다" | 학습 부담 | 주 1시간 학습 시간 공식 배정 |
| "기존 방식이 충분하다" | 변화 거부 | 수치 비교 (같은 작업 AI 유무 비교) |

### 무리하지 않는 도입 원칙

```
1. 강제하지 않기
   → "써야 한다" 대신 "써볼 수 있다"로 시작

2. 챔피언 활용하기
   → 먼저 잘 쓰는 사람이 팀 내 사례 공유

3. 작은 성공부터
   → "프로젝트 전체를 AI로" 가 아니라 "테스트 코드부터"

4. 실패 허용하기
   → AI 생성 코드가 리뷰에서 거절되어도 괜찮음

5. 정기적 피드백
   → 2주 회고에서 뭐가 되고 안 되는지 공유
```

## Step 5: CLAUDE.md를 팀 표준으로

AI 코딩 도구의 일관성은 프로젝트 설정 파일에서 시작해요.

### 팀용 CLAUDE.md 구조

```markdown
# CLAUDE.md

## 프로젝트 개요
{프로젝트 한 줄 설명}

## 기술 스택
- Frontend: {framework} + {library}
- Backend: {language} + {framework}
- DB: {database}

## 코딩 컨벤션
- {팀에서 합의한 스타일 규칙}
- {네이밍, 파일 구조, 에러 처리 패턴}

## AI 사용 규칙
- {허용/제한 범위}
- {PR 리뷰 기준}

## 자주 쓰는 명령어
- {빌드, 테스트, 린트 명령어}
```

핵심: CLAUDE.md는 **신규 팀원 온보딩 문서**이기도 해요. AI 도구가 아니라 사람이 읽어도 유용하게 작성하세요.

### 도구별 설정 파일 동기화

```
프로젝트 루트/
├── CLAUDE.md                         # Claude Code
├── .cursorrules                      # Cursor
├── .github/copilot-instructions.md   # GitHub Copilot
├── .windsurfrules                    # Windsurf
└── .ai-guidelines.md                 # 공통 가이드라인 (선택)
```

하나의 마스터 파일에서 나머지를 자동 생성하는 스크립트를 만들면 유지보수가 쉬워요.

## Step 6: 보안과 컴플라이언스

팀 도입에서 가장 자주 막히는 관문이 보안이에요.

### 보안 체크리스트

| 항목 | 체크 |
|------|------|
| 코드가 외부 서버로 전송되는가? | 기업용 플랜(Zero Data Retention) 확인 |
| API 키/시크릿이 AI 입력에 포함될 수 있는가? | `.env` 파일 AI 컨텍스트 제외 설정 |
| 생성된 코드의 라이선스는? | 도구별 이용약관 검토 |
| 코드 리뷰 없이 배포 가능한가? | PR 리뷰 필수 설정 (branch protection) |
| 감사 로그가 남는가? | AI 도구 사용 이력 추적 가능 여부 |

### 기업용 추천 설정

```yaml
# 보안 관련 도구 설정 예시
security:
  data_retention: zero          # 코드 학습에 사용 안 함
  context_exclude:
    - .env
    - .env.local
    - secrets/
    - credentials/
  branch_protection:
    require_pr: true
    required_reviewers: 2       # AI 코드 포함 시
    require_ci: true
```
