# 가이드 24: 프롬프트 캐싱과 컨텍스트 최적화

> AI 코딩 도구를 쓸수록 비용이 늘어난다면, 프롬프트 캐싱과 컨텍스트 관리를 점검할 때예요

## 소요 시간

15-25분

## 사전 준비

- AI 코딩 도구 사용 경험 (Claude Code, Cursor, Copilot 등)
- API 기반 LLM 사용 경험 또는 관심
- 토큰과 컨텍스트 윈도우 기본 개념

## 왜 프롬프트 캐싱이 중요한가요?

AI 코딩 도구를 매일 쓰면 토큰 사용량이 빠르게 늘어나요. 특히 큰 프로젝트에서 같은 파일을 반복적으로 읽히거나, CLAUDE.md 같은 설정 파일을 매 세션마다 다시 보내면 비용이 불필요하게 쌓여요.

```
일반적인 세션 흐름:
요청 1: CLAUDE.md(2K) + 소스파일(5K) + 질문(200) = 7,200 토큰
요청 2: CLAUDE.md(2K) + 소스파일(5K) + 이전 대화(3K) + 질문(300) = 10,300 토큰
요청 3: CLAUDE.md(2K) + 소스파일(5K) + 이전 대화(8K) + 질문(200) = 15,200 토큰

→ 3번의 요청에 CLAUDE.md만 6K 토큰을 중복 소비
```

프롬프트 캐싱은 이 중복을 해결해요. 이미 보낸 프롬프트의 앞부분이 캐시에 남아있으면, 캐시 히트된 부분은 입력 비용의 10%만 과금돼요.

| 구분 | 일반 입력 | 캐시 히트 | 절감률 |
|------|----------|----------|--------|
| Claude Sonnet 4 | $3/MTok | $0.30/MTok | 90% |
| Claude Opus 4 | $15/MTok | $1.50/MTok | 90% |
| Claude Haiku 3.5 | $0.80/MTok | $0.08/MTok | 90% |

## Step 1: 프롬프트 캐싱의 동작 원리

프롬프트 캐싱은 대화의 "접두사(prefix)"를 기준으로 동작해요. 매번 보내는 메시지에서 앞부분이 이전 요청과 동일하면, 그 부분은 캐시에서 가져와요.

### 캐시가 적용되는 조건

```
[시스템 프롬프트] + [CLAUDE.md] + [프로젝트 파일들] + [대화 이력] + [새 질문]
└──────── 캐시 가능 영역 ────────┘  └── 매번 변경 ──┘
```

캐시가 동작하려면 두 가지 조건이 필요해요:

1. **접두사 일치**: 메시지 배열의 앞부분이 이전 요청과 정확히 같아야 해요
2. **최소 길이**: 캐시 가능한 최소 토큰 수가 있어요 (Claude는 1,024 토큰 이상)

### 캐시 TTL (Time to Live)

| 제공자 | 기본 TTL | 비고 |
|--------|---------|------|
| Anthropic (Claude) | 5분 | 사용할 때마다 TTL 리셋 |
| OpenAI | 요청 간 자동 | 동일 접두사 자동 감지 |

Claude의 경우 5분 이내에 다음 요청을 보내면 캐시가 유지되고, 연속 작업 시 비용 절감이 커져요.

## Step 2: AI 코딩 도구별 캐싱 활용법

### Claude Code (CLI)

Claude Code는 프롬프트 캐싱을 자동으로 활용해요. 세션 내에서 연속 작업을 하면 이전 컨텍스트가 캐시돼요.

```bash
# 좋은 패턴: 한 세션에서 연속 작업
claude "프로젝트 구조 분석해줘"
# → 시스템 프롬프트 + CLAUDE.md 캐시됨

claude "이어서 API 엔드포인트 목록 정리해줘"
# → 앞부분 캐시 히트, 추가 질문만 새로 과금

# 비효율적 패턴: 세션을 매번 새로 시작
claude --no-cache "구조 분석해줘"
# (새 세션)
claude --no-cache "API 정리해줘"
# → 매번 전체 컨텍스트를 다시 보냄
```

**CLAUDE.md 최적화 팁:**

CLAUDE.md가 크면 캐시 효과가 커져요. 하지만 너무 크면 초기 비용이 늘어나니 적절한 크기가 중요해요.

```markdown
# 좋은 CLAUDE.md 구조 (캐시 친화적)

## 프로젝트 개요 (변경 빈도: 낮음) ← 캐시에 오래 남음
- 기술 스택, 아키텍처 패턴

## 코딩 규칙 (변경 빈도: 낮음) ← 캐시에 오래 남음
- 네이밍 컨벤션, 에러 처리 패턴

## 현재 작업 컨텍스트 (변경 빈도: 높음) ← 맨 아래 배치
- 이번 스프린트 목표, 진행 중인 이슈
```

핵심은 **변하지 않는 내용을 앞에, 자주 바뀌는 내용을 뒤에** 배치하는 거예요.

### Cursor

Cursor는 에디터 내에서 자동으로 컨텍스트를 관리해요. 파일 탭에 열어둔 파일이 컨텍스트로 들어가는데, 관련 파일만 열어두면 토큰 낭비를 줄일 수 있어요.

```
효과적인 작업 방식:
1. 작업할 파일 2-3개만 탭에 열기
2. @file로 필요한 파일 명시적 참조
3. 불필요한 탭 닫기 → 자동 컨텍스트에서 제외

비효율적인 방식:
1. 탭 20개 열어놓기 → 모든 파일이 컨텍스트에 포함
2. "전체 코드베이스 보고 고쳐줘" → 토큰 폭발
```

### API 직접 사용 시

API를 직접 호출할 때는 캐시 브레이크포인트를 명시적으로 설정할 수 있어요.

```python
import anthropic

client = anthropic.Anthropic()

# 캐시할 시스템 프롬프트에 cache_control 설정
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "당신은 시니어 백엔드 개발자입니다...(긴 시스템 프롬프트)",
            "cache_control": {"type": "ephemeral"}
        }
    ],
    messages=[
        {"role": "user", "content": "API 설계 리뷰해줘"}
    ]
)

# 응답의 usage에서 캐시 상태 확인
print(f"캐시 생성: {response.usage.cache_creation_input_tokens}")
print(f"캐시 히트: {response.usage.cache_read_input_tokens}")
```

## Step 3: 컨텍스트 윈도우 최적화 전략

프롬프트 캐싱만으로는 부족해요. 컨텍스트 윈도우 자체를 효율적으로 쓰는 전략이 필요해요.

### 3-1. 계층적 컨텍스트 로딩

모든 파일을 한 번에 넣지 말고, 필요할 때 불러오는 패턴이에요.

```
Level 1 (항상 포함): CLAUDE.md, 프로젝트 규칙
Level 2 (작업 시 포함): 현재 수정 중인 파일 2-3개
Level 3 (필요 시 참조): 관련 테스트, 타입 정의
Level 4 (요청 시만): 전체 디렉토리 구조, 의존성 트리
```

```bash
# Claude Code에서 계층적 컨텍스트 사용 예시
claude "src/api/users.ts 파일만 보고 타입 에러 수정해줘"
# → Level 1 + Level 2만 사용

# 필요하면 추가 컨텍스트 요청
claude "관련 테스트 파일도 확인해서 테스트도 수정해줘"
# → Level 3 추가
```

### 3-2. 컨텍스트 컴팩션

긴 대화가 이어지면 이전 내용을 요약해서 컨텍스트를 줄이는 방법이에요.

```
세션 초반 (컨텍스트 여유로움):
├── 전체 대화 이력 포함
├── 코드 변경 사항 전부 기록
└── AI 추론 과정도 유지

세션 중반 (컨텍스트 70% 이상):
├── 핵심 결정사항만 유지
├── 코드 변경은 diff만 보관
└── 불필요한 탐색 기록 제거

세션 후반 (컨텍스트 90% 이상):
├── 새 세션 시작 권장
├── 핸드오프 메모 작성
└── 결과물만 정리해서 전달
```

Claude Code에서는 `/compact` 명령으로 컨텍스트를 정리할 수 있어요:

```bash
# 컨텍스트가 많이 쌓였을 때
> /compact

# 특정 관점으로 요약 요청
> /compact API 리팩토링 관련 내용만 유지해줘
```

### 3-3. 파일 참조 최적화

AI에게 전체 파일을 보여주는 대신, 필요한 부분만 보여주면 토큰을 크게 아낄 수 있어요.

| 패턴 | 토큰 | 효과 |
|------|------|------|
| `파일 전체 보여줘` | ~2,000 | 불필요한 import, 주석 포함 |
| `함수 X만 보여줘` | ~300 | 핵심 로직만 포함 |
| `타입 정의만 보여줘` | ~150 | 인터페이스만 포함 |

```bash
# 효과적인 참조 패턴
claude "src/api/users.ts의 createUser 함수에서 validation 로직 리뷰해줘"

# 비효율적인 참조 패턴
claude "src/ 폴더 전체 읽고 문제 찾아줘"
```

## Step 4: 실전 비용 절감 체크리스트

지금 바로 적용할 수 있는 최적화 목록이에요.

### 세션 관리

| 체크 | 항목 | 예상 절감 |
|------|------|----------|
| ☐ | 관련 작업은 한 세션에서 연속으로 수행 | 30-50% |
| ☐ | 세션 시작 시 작업 범위를 명확하게 선언 | 15-20% |
| ☐ | 컨텍스트 70% 이상이면 `/compact` 실행 | 20-30% |
| ☐ | 전혀 다른 작업은 새 세션으로 분리 | 10-15% |

### CLAUDE.md 최적화

| 체크 | 항목 | 효과 |
|------|------|------|
| ☐ | 고정 규칙을 파일 상단에 배치 | 캐시 히트율 향상 |
| ☐ | 변동 컨텍스트를 파일 하단에 배치 | 캐시 무효화 최소화 |
| ☐ | 불필요한 예시/설명 제거 (간결하게) | 토큰 절약 |
| ☐ | .claudeignore로 불필요한 파일 제외 | 자동 컨텍스트 절약 |

### API 사용 시

| 체크 | 항목 | 효과 |
|------|------|------|
| ☐ | 시스템 프롬프트에 `cache_control` 설정 | 90% 입력 비용 절감 |
| ☐ | 긴 참조 문서는 시스템 메시지에 포함 | 캐시 대상이 됨 |
| ☐ | 응답의 `cache_read_input_tokens` 모니터링 | 캐시 효과 추적 |

## Step 5: MCP 도구와 컨텍스트 관리

MCP 서버를 많이 연결하면 각 서버의 도구 설명이 컨텍스트를 차지해요. 도구가 20개면 도구 설명만으로 수천 토큰이 사라질 수 있어요.

### MCP 컨텍스트 줄이기

```json
// mcp-servers.json — 필요한 서버만 프로젝트별로 설정
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/project"]
    }
    // ❌ 사용하지 않는 서버는 제거
    // "slack": { ... },
    // "notion": { ... }
  }
}
```

| 전략 | 설명 |
|------|------|
| **프로젝트별 MCP 설정** | 프로젝트마다 필요한 도구만 활성화 |
| **도구 설명 간소화** | 커스텀 MCP 서버 제작 시 description을 짧게 |
| **레이지 로딩** | 도구를 미리 전부 로드하지 않고 필요 시 호출 |

## 핵심 정리

```
프롬프트 캐싱 3원칙:
1. 고정 내용은 앞에, 변동 내용은 뒤에
2. 한 세션에서 연속 작업 → 캐시 히트 극대화
3. 컨텍스트 모니터링 → 70% 넘으면 정리

비용 절감 우선순위:
1순위: 세션 관리 (무료, 즉시 적용)
2순위: CLAUDE.md 구조 최적화 (30분 투자)
3순위: API cache_control 설정 (개발 필요)
4순위: MCP 서버 정리 (프로젝트별 10분)
```
