# 23. AI 코드 테스트 전략 가이드

> AI 에이전트가 생성한 코드를 어떻게 테스트해야 할까? — 테스트 프롬프팅부터 커버리지 확보, 엣지케이스 유도까지

## 왜 AI 코드 테스트가 다른가요?

AI가 코드를 생성하면 "동작하는 것처럼 보이는 코드"가 나와요. 컴파일도 되고 기본 케이스도 통과하죠. 하지만 문제는 보이지 않는 곳에 있어요:

- 경계값(boundary)에서 예상치 못한 동작
- 에러 핸들링이 빠져있거나 너무 광범위한 try-catch
- 비즈니스 로직의 미묘한 차이를 AI가 놓치는 경우
- "그럴듯한" 코드가 실제로는 잘못된 알고리즘을 사용하는 경우

그래서 AI 코드에 대한 테스트 전략은 일반적인 TDD와는 관점이 달라야 해요.

---

## 전략 1: 테스트 먼저, 코드 나중에 (Test-First Prompting)

AI에게 코드를 요청하기 전에, 먼저 테스트를 작성하게 하세요.

```
이 함수의 테스트 케이스를 먼저 작성해줘.
요구사항: 사용자 이름 유효성 검사 함수
- 2~20자 한글/영문/숫자만 허용
- 특수문자, 공백 불가
- 빈 문자열은 거부
```

테스트가 나오면 검토하고, 그 다음에 구현을 요청해요:

```
위 테스트를 모두 통과하는 구현을 작성해줘.
```

### 왜 이렇게 하나요?

| 방식 | 문제 |
|------|------|
| 코드 먼저 → 테스트 나중 | AI가 자기 코드에 맞는 테스트만 만듦 (순환 검증) |
| 테스트 먼저 → 코드 나중 | 테스트가 독립적인 검증 기준이 됨 |

핵심은 **테스트와 구현을 분리**하는 거예요. 같은 프롬프트에서 동시에 요청하면 AI가 자기 코드를 정당화하는 테스트를 만들어요.

---

## 전략 2: 엣지케이스 유도 프롬프트

AI는 해피 패스(happy path)에 집중하는 경향이 있어요. 엣지케이스를 명시적으로 요청해야 해요.

### 프롬프트 패턴

```
이 함수에 대해 다음 범주별로 테스트를 작성해줘:

1. 정상 동작 (happy path) — 3개
2. 경계값 (boundary) — 빈 배열, 최대 크기, 0, -1
3. 에러 케이스 — null, undefined, 잘못된 타입
4. 동시성 — 동시 호출 시 race condition
5. 성능 — 대량 데이터 (10만 건) 처리 시간
```

### 범주별 체크리스트

| 범주 | 확인 항목 | 예시 |
|------|----------|------|
| 경계값 | 빈 입력, 최대/최소값, 0 | `[]`, `""`, `Number.MAX_SAFE_INTEGER` |
| 타입 | 잘못된 타입, null, undefined | `null`, `42` (문자열 기대 시) |
| 상태 | 초기 상태, 중간 상태, 종료 상태 | 로딩 중 재호출, 이미 닫힌 연결 |
| 동시성 | 병렬 호출, 순서 의존성 | 동시 업데이트, 이벤트 순서 역전 |
| 외부 의존 | 네트워크 실패, 타임아웃, 느린 응답 | API 503, 30초 지연 |

## 전략 3: 뮤테이션 테스팅으로 테스트 품질 검증

AI가 생성한 테스트가 정말 유효한지 확인하는 방법이에요.

```
다음 코드에서 의도적으로 버그를 3개 심어줘.
원본 코드는 유지하고, 버그 버전을 따로 만들어줘.
버그마다 어떤 테스트가 잡아야 하는지도 알려줘.
```

**검증 과정:**

1. 정상 코드로 모든 테스트 통과 확인
2. 버그 코드로 테스트 실행 → 실패해야 정상
3. 테스트가 버그를 못 잡으면 → 테스트가 약한 것

```bash
# 예시: Jest로 뮤테이션 테스팅 자동화
npx stryker run
```

| 결과 | 의미 | 조치 |
|------|------|------|
| 뮤턴트 killed | 테스트가 버그를 잡음 | 좋은 테스트 |
| 뮤턴트 survived | 테스트가 버그를 놓침 | 테스트 보강 필요 |
| 타임아웃 | 무한 루프 발생 | 가드 조건 추가 |

## 전략 4: 계층별 테스트 전략

AI 코드에도 테스트 피라미드를 적용하되, 비율을 조정해요.

```
테스트 피라미드 (AI 코드 버전):

        ┌──────────┐
        │  E2E (5%)│  ← 핵심 플로우만
        ├──────────┤
        │통합 (35%)│  ← AI 코드의 약점 집중
        ├──────────┤
        │단위 (60%)│  ← 로직 검증
        └──────────┘
```

### AI 코드에서 통합 테스트 비율이 높은 이유

AI가 작성한 코드는 개별 함수 단위로는 정상이지만 **함수 간 연결에서 문제**가 자주 발생해요:

- 타입 불일치 (함수 A의 출력과 함수 B의 입력)
- 상태 전파 누락 (에러가 상위로 전달되지 않음)
- 사이드 이펙트 미반영 (DB 업데이트 후 캐시 무효화 누락)

```typescript
// 통합 테스트 예시: 주문 생성 플로우 전체 검증
describe('Order Creation Flow', () => {
  it('creates order and updates inventory', async () => {
    // Given: 재고 10개인 상품
    const product = await createProduct({ stock: 10 });
    
    // When: 3개 주문
    const order = await createOrder({ 
      productId: product.id, 
      quantity: 3 
    });
    
    // Then: 주문 생성 + 재고 감소 + 이벤트 발행
    expect(order.status).toBe('confirmed');
    
    const updated = await getProduct(product.id);
    expect(updated.stock).toBe(7);  // ← AI가 자주 놓치는 부분
    
    const events = await getEvents(order.id);
    expect(events).toContainEqual(
      expect.objectContaining({ type: 'ORDER_CREATED' })
    );
  });
});
```

## 전략 5: AI 코드 리뷰 + 테스트 보강 루프

코드 생성과 테스트를 반복적으로 강화하는 워크플로우예요.

### 3단계 루프

**1단계: 초기 생성**
```
이 API 엔드포인트를 구현하고 테스트를 작성해줘.
```

**2단계: 비판적 리뷰 요청**
```
방금 작성한 코드에서 잠재적 버그나 빠진 엣지케이스를 찾아줘.
특히 에러 핸들링, 입력 검증, 동시성 문제를 집중적으로 봐줘.
```

**3단계: 보강 테스트 추가**
```
위에서 발견한 문제들을 검증하는 테스트를 추가해줘.
```

### 프롬프트 팁

| 상황 | 프롬프트 |
|------|---------|
| 테스트가 너무 단순할 때 | "이 테스트들 중 삭제해도 커버리지에 영향 없는 게 있어?" |
| 모킹이 과도할 때 | "이 모킹을 실제 구현으로 바꾸면 뭐가 깨져?" |
| 비즈니스 로직 검증 | "이 함수가 돈 계산에 쓰인다고 가정하고 테스트 강화해줘" |
| 성능 회귀 방지 | "이 함수의 실행 시간이 100ms를 넘지 않는지 검증해줘" |

## 전략 6: 커버리지 기반 보강

AI에게 커버리지 리포트를 보여주고 누락된 부분을 채워달라고 해요.

```bash
# 커버리지 리포트 생성
npx jest --coverage --coverageReporters=text

# 또는 Python
pytest --cov=src --cov-report=term-missing
```

```
이 커버리지 리포트를 보고 커버리지가 낮은 파일의 누락된 브랜치를 
테스트로 채워줘. 특히 uncovered lines를 집중적으로 다뤄줘.

[커버리지 리포트 붙여넣기]
```

### 커버리지 목표 가이드라인

| 영역 | 목표 | 이유 |
|------|------|------|
| 비즈니스 로직 | 90%+ | 돈, 데이터 무결성과 직결 |
| API 핸들러 | 80%+ | 입력 검증, 에러 응답 중요 |
| 유틸리티 | 95%+ | 순수 함수, 테스트 쉬움 |
| UI 컴포넌트 | 70%+ | 렌더링 + 인터랙션 핵심만 |
| 설정/글루 코드 | 50%+ | 통합 테스트에서 간접 검증 |

## 실전 워크플로우 요약

```
[Step 1] 요구사항 정리
    ↓
[Step 2] 테스트 먼저 생성 (AI)
    ↓
[Step 3] 테스트 직접 검토 + 엣지케이스 보강 요청
    ↓
[Step 4] 구현 코드 생성 (AI)
    ↓
[Step 5] 테스트 실행 → 실패 분석
    ↓
[Step 6] 비판적 리뷰 요청 → 보강 테스트
    ↓
[Step 7] 커버리지 확인 → 누락 브랜치 보강
    ↓
[Step 8] 뮤테이션 테스팅 (선택)
```

## 흔한 실수와 해결

| 실수 | 왜 문제인가 | 해결 |
|------|-----------|------|
| 코드와 테스트를 한 번에 요청 | 자기 코드 정당화 테스트 생성 | 분리해서 요청 |
| `expect(result).toBeTruthy()` 남발 | 구체적인 값 검증 안 됨 | 정확한 기대값 명시 |
| 모킹으로 모든 외부 의존 제거 | 통합 문제를 영원히 못 발견 | 통합 테스트에서 실제 연결 |
| AI가 만든 테스트를 그대로 신뢰 | 해피 패스만 검증되는 경우 많음 | 엣지케이스 수동 추가 |
| 100% 커버리지 추구 | 의미 없는 테스트 양산 | 위험도 기반 우선순위 |
