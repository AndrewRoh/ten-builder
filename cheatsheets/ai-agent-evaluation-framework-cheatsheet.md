# AI 코딩 에이전트 평가 프레임워크 치트시트

> AI 코딩 도구를 어떻게 비교하고 선택할 것인가 — 벤치마크 해석부터 실전 메트릭까지 한 페이지로 정리

## 주요 벤치마크 한눈에 보기

| 벤치마크 | 측정 대상 | 특징 | 주의점 |
|----------|----------|------|--------|
| **SWE-bench Verified** | 실제 GitHub Issue 해결률 | 500개 검증된 태스크, 가장 널리 인용 | Harness 설계에 따라 점수 차이 큼 |
| **SWE-bench Pro** | 고난이도 SE 태스크 | 731개 프론티어 태스크, 가중치 50% | Verified보다 점수가 20~30%p 낮음 |
| **LiveCodeBench** | 경쟁 프로그래밍 | 오염 없는 새 문제, 주기적 갱신 | 알고리즘 위주라 실무와 다를 수 있음 |
| **HumanEval / MBPP** | 함수 단위 코드 생성 | 초기 벤치마크, 이해하기 쉬움 | 포화 상태 — 대부분 95%+ |
| **Aider Polyglot** | 멀티 언어 편집 정확도 | 다양한 언어 지원 검증 | 편집 포맷에 의존적 |

## 벤치마크 점수 올바르게 읽기

### 같은 벤치마크, 다른 점수가 나오는 이유

```
점수 = 모델 성능 × Harness 품질 × 실행 환경

예시: 같은 모델이라도
├── Harness A (검색 서브에이전트 포함) → 72%
├── Harness B (기본 프롬프트만)     → 58%
└── Harness C (반복 재시도 허용)     → 68%
```

### 체크리스트: 벤치마크 결과 검증

- [ ] **테스트 셋이 동일한가?** (Verified vs Lite vs Full)
- [ ] **Harness 코드가 공개되었는가?**
- [ ] **시도 횟수(attempts)가 명시되었는가?** (pass@1 vs pass@5)
- [ ] **비용/시간이 함께 보고되었는가?**
- [ ] **측정 시점이 최근인가?** (3개월 이상 지난 결과는 참고용)

## 실전 평가 메트릭 5가지

벤치마크만으로는 부족해요. 팀에서 직접 측정해야 할 핵심 지표:

| 메트릭 | 계산법 | 목표 기준 |
|--------|--------|----------|
| **첫 시도 성공률** | 수정 없이 통과한 태스크 / 전체 태스크 | > 60% |
| **태스크 완료 시간** | 에이전트 시작 → 최종 커밋 | 단순 태스크 < 5분 |
| **토큰 효율** | 소비 토큰 / 변경된 코드 라인 | < 500 토큰/LOC |
| **리트라이 비율** | 재시도가 필요했던 태스크 비율 | < 30% |
| **사람 개입률** | 수동 수정이 필요했던 비율 | < 20% |

### 측정 자동화 예시

```bash
# 에이전트 세션 로그에서 메트릭 추출
cat session-log.jsonl | jq '{
  total_tokens: .usage.total_tokens,
  lines_changed: .diff_stats.lines_added,
  attempts: .retry_count,
  duration_sec: (.end_ts - .start_ts),
  human_edit: .post_edit_required
}'
```

```python
# 주간 평가 스크립트
def calculate_efficiency(sessions):
    return {
        "first_try_rate": sum(1 for s in sessions if s["attempts"] == 1) / len(sessions),
        "avg_tokens_per_loc": sum(s["tokens"] for s in sessions) / max(sum(s["loc"] for s in sessions), 1),
        "avg_duration": sum(s["duration"] for s in sessions) / len(sessions),
        "human_intervention": sum(1 for s in sessions if s["human_edit"]) / len(sessions),
    }
```

## 태스크 난이도별 기대 성능

| 난이도 | 예시 | 기대 성공률 | 평균 시간 |
|--------|------|-----------|----------|
| **단순** | 변수명 변경, import 추가, 오타 수정 | 95%+ | < 30초 |
| **보통** | 함수 구현, 단위 테스트 작성, 버그 수정 | 70~85% | 1~5분 |
| **복잡** | API 엔드포인트 추가, 리팩토링, 마이그레이션 | 40~60% | 5~15분 |
| **고난이도** | 아키텍처 변경, 멀티 파일 연쇄 수정 | 20~40% | 15분+ |

## 비용 효율 평가 프레임워크

### 비용 대비 성능 비교

```
ROI 점수 = (첫 시도 성공률 × 태스크 복잡도 가중치) / 월 비용

예시:
├── 도구 A: (0.72 × 1.5) / $200 = 0.0054
├── 도구 B: (0.65 × 1.5) / $100 = 0.0098  ← 가성비 우수
└── 도구 C: (0.80 × 1.5) / $400 = 0.0030
```

| 비용 항목 | 측정 방법 |
|-----------|----------|
| 토큰 비용 | 입력 + 출력 토큰 × 단가 |
| 구독료 | 월 고정 비용 (Pro/Max 플랜) |
| 숨은 비용 | 실패 시 재시도 토큰, 수동 수정 시간 |
| 기회 비용 | 에이전트 대기 시간 동안 개발자 유휴 |

### 월 비용 산정 공식

```
월 예상 비용 = (일 평균 세션 수 × 세션당 평균 토큰) × 토큰 단가 × 22일
            + 월 구독료
            + (리트라이율 × 추가 토큰 비용)
```

## 평가 프로세스 3단계

### 1단계: 파일럿 테스트 (1주)

```yaml
# pilot-test-config.yaml
test_tasks:
  simple: 10    # 변수명 변경, import 추가
  medium: 10    # 함수 구현, 테스트 작성
  complex: 5    # API 추가, 리팩토링
  hard: 3       # 아키텍처 변경

metrics_to_track:
  - first_try_success_rate
  - avg_completion_time
  - tokens_consumed
  - human_intervention_count
```

### 2단계: A/B 비교 (2주)

같은 태스크를 두 도구에 동시 할당하여 직접 비교:

| 비교 항목 | 도구 A | 도구 B | 승자 |
|-----------|--------|--------|------|
| 성공률 | | | |
| 평균 시간 | | | |
| 토큰 비용 | | | |
| 코드 품질 | | | |

### 3단계: 팀 롤아웃 판단

```
채택 조건 = 성공률 > 60%
          AND 비용 < 개발자 시급 × 절약 시간
          AND 팀 만족도 > 7/10
```

## 흔한 평가 실수

| 실수 | 왜 문제인가 | 해결 |
|------|-----------|------|
| 벤치마크 점수만 보고 결정 | 실무 환경과 벤치마크 조건이 다름 | 자체 태스크셋으로 파일럿 |
| Harness 차이 무시 | 같은 모델이 30%p 차이 날 수 있음 | Harness 조건 함께 확인 |
| 비용 미포함 비교 | 성능 좋지만 3배 비쌀 수 있음 | ROI 점수로 정규화 |
| pass@1만 측정 | 실제로는 재시도 포함 성공률이 중요 | pass@1과 pass@3 모두 기록 |
| 단순 태스크만 테스트 | 단순 태스크는 대부분 비슷한 결과 | 복잡 태스크 비중 30% 이상 |
| 측정 기간 너무 짧음 | 1~2일로는 신뢰 구간 부족 | 최소 1주, 태스크 25개 이상 |
