# AI 모델 라우팅 치트시트

> 작업 유형별 최적 AI 모델 선택 — 한 페이지 요약

## 왜 모델 라우팅이 필요한가?

하나의 모델로 모든 코딩 작업을 처리하면 비용은 올라가고 속도는 느려져요. 간단한 리팩토링에 최상위 모델을 쓸 필요가 없고, 복잡한 아키텍처 설계에 경량 모델을 쓰면 결과가 부실해요. 작업 성격에 맞는 모델을 골라쓰는 게 2026년 AI 코딩의 기본이에요.

## 모델 티어 분류

| 티어 | 대표 모델 | 특징 | 토큰 비용 (입/출) |
|------|----------|------|-------------------|
| **경량 (Fast)** | Claude Haiku, GPT-4o mini, Gemini Flash | 낮은 지연, 저비용 | ~$0.25 / $1.25 per 1M |
| **범용 (Standard)** | Claude Sonnet, GPT-4o, Gemini Pro | 속도·품질 균형 | ~$3 / $15 per 1M |
| **고급 (Premium)** | Claude Opus, GPT-5, Gemini Ultra | 최고 추론 능력 | ~$15 / $75 per 1M |
| **추론 특화** | o3, Claude Opus + Extended Thinking | 복잡한 논리 체인 | 가변 (추론 토큰 별도) |

> 가격은 2026년 3월 기준 대략적인 범위예요. 실제 가격은 각 서비스 페이지를 확인하세요.

## 작업별 라우팅 가이드

### 경량 모델을 쓰세요

| 작업 | 이유 | 예시 프롬프트 |
|------|------|-------------|
| 변수명·함수명 변경 | 패턴이 단순하고 반복적 | `"이 파일의 camelCase를 snake_case로 바꿔줘"` |
| 코드 포매팅 | 규칙 기반 변환 | `"Prettier 규칙에 맞게 정리해줘"` |
| 단순 타입 추가 | 기존 코드에서 추론 가능 | `"이 JS 파일에 TypeScript 타입을 추가해줘"` |
| 보일러플레이트 생성 | 정해진 패턴 반복 | `"Express 라우터 CRUD 보일러플레이트 만들어줘"` |
| 주석·문서 작성 | 기존 코드 설명 | `"이 함수에 JSDoc 주석을 달아줘"` |
| 에러 메시지 번역 | 직역에 가까운 작업 | `"에러 메시지를 한국어로 바꿔줘"` |

### 범용 모델을 쓰세요

| 작업 | 이유 | 예시 프롬프트 |
|------|------|-------------|
| 함수·클래스 구현 | 문맥 파악 + 로직 작성 | `"이 인터페이스에 맞는 서비스 클래스를 구현해줘"` |
| 버그 수정 | 원인 분석 + 코드 수정 | `"이 테스트가 실패하는 이유를 찾고 수정해줘"` |
| 단위 테스트 작성 | 엣지케이스 고려 필요 | `"이 유틸 함수의 테스트를 jest로 작성해줘"` |
| API 엔드포인트 추가 | 여러 파일 수정 | `"사용자 프로필 업데이트 API를 추가해줘"` |
| 코드 리뷰 | 패턴 인식 + 개선 제안 | `"이 PR의 코드를 리뷰해줘"` |
| DB 쿼리 최적화 | 실행 계획 이해 필요 | `"이 느린 쿼리를 최적화해줘"` |

### 고급 모델을 쓰세요

| 작업 | 이유 | 예시 프롬프트 |
|------|------|-------------|
| 아키텍처 설계 | 전체 구조 이해 + 트레이드오프 분석 | `"마이크로서비스로 전환할 때 서비스 경계를 설계해줘"` |
| 대규모 리팩토링 계획 | 영향 범위 파악 + 단계 분할 | `"이 모노리스를 모듈로 분리하는 마이그레이션 계획을 세워줘"` |
| 보안 감사 | 다양한 공격 벡터 추론 | `"이 인증 플로우의 보안 취약점을 분석해줘"` |
| 복잡한 알고리즘 | 수학적 추론 + 최적화 | `"이 추천 알고리즘의 시간 복잡도를 O(n log n)으로 개선해줘"` |
| 레거시 시스템 분석 | 문서 없는 코드 역공학 | `"이 10년된 코드베이스의 아키텍처를 파악하고 문서화해줘"` |

## 라우팅 의사결정 플로우

```
작업 받음
  │
  ├─ 패턴이 단순하고 반복적? ──→ 경량 모델
  │
  ├─ 여러 파일을 읽어야 하지만 ──→ 범용 모델
  │  로직은 표준적?
  │
  ├─ 전체 아키텍처를 이해해야 ──→ 고급 모델
  │  하거나 트레이드오프 분석?
  │
  └─ 수학적 추론이나 ──→ 추론 특화 모델
     복잡한 논리 체인?
```

## 실전 라우팅 설정 예시

### Claude Code에서 모델 전환

```bash
# 기본 작업 — Sonnet으로 시작
claude --model sonnet "이 함수의 테스트를 작성해줘"

# 복잡한 작업 — Opus로 전환
claude --model opus "이 시스템의 아키텍처를 분석하고 개선안을 제시해줘"

# 단순 반복 — Haiku로 비용 절약
claude --model haiku "이 파일들의 import 순서를 정리해줘"

# 세션 중 모델 변경
/model sonnet    # 대화 중 Sonnet으로 전환
/model opus      # 복잡해지면 Opus로 변경
```

### API 레벨 라우팅 (자동화 스크립트)

```python
def select_model(task_type: str, file_count: int) -> str:
    """작업 유형과 파일 수로 모델을 자동 선택"""
    
    if task_type in ["format", "rename", "comment", "boilerplate"]:
        return "claude-haiku"
    
    if task_type in ["implement", "test", "bugfix", "review"]:
        return "claude-sonnet"
    
    if task_type in ["architecture", "security", "migration"]:
        return "claude-opus"
    
    # 파일 수가 많으면 한 단계 올림
    if file_count > 20:
        return "claude-opus"
    if file_count > 5:
        return "claude-sonnet"
    
    return "claude-sonnet"  # 기본값
```

## 비용 절감 효과

| 시나리오 | 단일 모델 (Opus만) | 라우팅 적용 | 절감율 |
|----------|-------------------|------------|--------|
| 일일 100 태스크 | ~$45 | ~$12 | **73%** |
| 주간 코드 리뷰 20건 | ~$30 | ~$15 | **50%** |
| 리팩토링 프로젝트 | ~$120 | ~$55 | **54%** |

> 실제 절감 효과는 작업 구성에 따라 달라져요. 경량 작업 비율이 높을수록 절감 폭이 커요.

## 흔한 실수

| 실수 | 왜 문제인가 | 해결 |
|------|-----------|------|
| 모든 작업에 Opus 사용 | 비용 10배, 속도 3배 느림 | 작업별 티어 분류 후 라우팅 |
| 아키텍처 설계에 Haiku 사용 | 얕은 분석, 중요한 트레이드오프 누락 | 설계·보안·분석은 고급 모델 |
| 모델 전환 없이 긴 세션 유지 | 컨텍스트 낭비 + 비용 누적 | 작업 전환 시 모델도 전환 |
| 추론 모델로 단순 코딩 | 느리고 비싸며 과잉 분석 | 추론은 복잡한 논리 문제에만 |
| 비용만 보고 항상 경량 선택 | 결과 품질 저하로 재작업 증가 | 품질 기준선 이상이면 경량, 아니면 올림 |

## 빠른 판단 기준

```
"5분이면 끝나는 작업"     → 경량 모델
"30분 정도 걸리는 작업"    → 범용 모델
"하루 이상 고민할 작업"    → 고급 모델
"정답이 하나인 작업"       → 경량 모델
"트레이드오프가 있는 작업"  → 고급 모델
```
